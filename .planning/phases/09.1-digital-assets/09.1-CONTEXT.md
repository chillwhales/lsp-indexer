<decisions>
## Implementation Decisions

### Data Richness — Full rich type with include toggles (inverted default)

- **Same architectural pattern as profiles**: every field is controllable via `@include(if:)` directives in GraphQL documents
- **Inverted default vs profiles**: when `include` is `undefined`, fetch **everything** — opt-out rather than opt-in. Only when `include` is explicitly provided do the toggles take effect
- **Only `address` is guaranteed** — always fetched regardless of include state
- **All other fields are includable**: name, symbol, tokenType, totalSupply, decimals, description, category, icons, images, links, attributes, owner, holderCount, creatorCount, LSP8-specific fields
- **LSP8-specific fields included** in the type as nullable — `referenceContract`, `tokenIdFormat`, `baseUri` — a digital asset can be either LSP7 or LSP8, so these fields belong on `DigitalAsset`
- **Owner field**: `owner: { address: string, timestamp: string } | null` — contract controller, not token holder — behind include toggle

### Derived `standard` field — LSP7 vs LSP8 disambiguation

- **Parser derives `standard: "LSP7" | "LSP8"`** from presence of the `decimals` field (`decimals` only exists on LSP7 contracts, never LSP8)
- **4 distinct token cases** from combining `standard` + `tokenType`:
  - LSP7 Token (`standard: "LSP7"`, `tokenType: 0`)
  - LSP7 NFT (`standard: "LSP7"`, `tokenType: 1`)
  - LSP8 NFT (`standard: "LSP8"`, `tokenType: 1`)
  - LSP8 Collection (`standard: "LSP8"`, `tokenType: 2`)
- **`standard` is a first-class field on `DigitalAsset`** — exposed publicly, not just internal to the playground

### Token Type Filter — Use `@lukso/lsp4-contracts` constants

- **Import `LSP4_TOKEN_TYPES` from `@lukso/lsp4-contracts`** for filter values — no raw `"0"`/`"1"`/`"2"` strings exposed to consumers
- Service layer maps constants to Hasura raw values internally

### Filter Fields

- `name` — `_ilike` (case-insensitive)
- `symbol` — `_ilike`
- `tokenType` — exact match using `LSP4_TOKEN_TYPES` constants
- `category` — `_ilike` (LSP4 metadata category, free-text, case-insensitive)
- `holderAddress` — `_ilike` (address that holds tokens of this asset — same as owned_assets but kept here for hook flexibility)
- `ownerAddress` — `_ilike` (contract controller address)
- **Philosophy**: hooks should be flexible — don't force users to switch hooks for adjacent data. Domain boundaries are guidelines, not restrictions.

### Sort Fields

- `name`, `symbol`, `holderCount`, `creatorCount`, `totalSupply`, `createdAt`
- **Note for researcher/planner**: verify how `createdAt` is surfaced in the Hasura `digital_asset` schema (may be via `owner.timestamp` or a different field — needs confirmation before implementation)

### Playground Card — Full card, color-coded badges

- **List card shows**: icon (first LSP4 icon image), name, symbol, `standard`+`tokenType` color-coded badge, description snippet, holderCount, creatorCount, totalSupply, owner address (truncated)
- **Color-coded badges** for 4 derived types:
  - LSP7 Token → blue
  - LSP7 NFT → purple
  - LSP8 NFT → orange
  - LSP8 Collection → yellow
- **Single asset tab shows**: full card + LSP4 metadata section (description, category, attributes array, links array) + conditional LSP8 section (referenceContract, tokenIdFormat, baseUri) only when `standard === "LSP8"`
- **Preset addresses** (2):
  - CHILL: `0x5B8B0E44D4719F8A328470DcCD3746BFc73d6B14` (LSP7)
  - Chillwhales: `0x86E817172b5c07f7036Bf8aA46e2db9063743A83` (LSP8 collection)

<specifics>
## Specific Ideas

- "Same approach as profiles" — the include toggle pattern, parser architecture, and hook API shape all mirror Phase 8. When in doubt, look at how profiles does it.
- The `standard` field derivation (`decimals` presence → LSP7 vs LSP8) is a key SDK contribution — LUKSO's own tooling doesn't always make this easy to determine.
- `holderAddress` and `ownerAddress` as separate filter fields deliberately — contract controller vs token holder are different concepts, both useful.
- Reference implementation exists on `feat/phase-9-query-domains` branch (digital-assets files) — use as reference for Hasura query structure, but the type shape and include pattern defined here supersedes it.

</specifics>

## Deferred Ideas

- **Profile hooks include default fix**: Profile hooks (`useProfiles`, `useInfiniteProfiles`) currently default include to opt-in (undefined = fetch nothing extra). User wants profiles to also default to fetch-everything when include is undefined — same as digital assets. This is a fix to Phase 8 and should be addressed as a discovered todo in a future session (or at the start of Phase 9.1 if trivial).
