## User Constraints (from CONTEXT.md)

### Locked Decisions

**Strictness of excluded fields:**

- Excluded fields are **absent from the type** — accessing them is a TypeScript error, not `undefined` or `null`
- `include: { name: true, tags: false }` and `include: { name: true }` produce the **same narrowed type** — `false` and omission are identical
- `include: {}` (empty object) returns **base fields only** — it means "I opted in to nothing." Omitting `include` entirely returns the full type
- Parsers **strip excluded fields at runtime** — `Object.keys()` only shows included fields, runtime matches the type. Not type-level-only

**Always-present base fields:**

- **`address` only** as the always-present base field — every domain, every nesting level. Everything else (scalars, counts, relations) is opt-in through `include`
- **Derived fields follow their source** — `standard` is present only when `decimals` is included (since it's derived from it). `tokenType` gets its own include toggle. No magic fields appearing without their source data
- **Aggregate/count fields** (`followerCount`, `holderCount`, `creatorCount`) are **regular include fields** — no special treatment, no grouping
- **Relations** follow the same rules as scalars — not listed in include = absent from type = not fetched

**Nested include depth:**

- `true` is shorthand for **"give me the full type"** on relations — `include: { digitalAsset: true }` returns complete DigitalAsset. Sub-include object narrows: `include: { digitalAsset: { name: true, symbol: true } }`
- Type narrowing goes **as deep as the existing Zod include schemas** — currently up to 3 levels (OwnedToken → NFT → Collection). No artificial limit, no generic recursive solution for depths that don't exist
- **Same base-field rules at every nesting level** — nested `digitalAsset` narrowed with `{ name: true }` gives `{ address, name }`. Consistent, one mental model
- **Shared sub-include types** — `DigitalAssetInclude` is reused wherever a digital asset relation appears (OwnedAsset, NFT collection, etc.). Existing exception: `OwnedTokenNftInclude` omits `collection`/`holder` from `NftInclude` — that stays

**Playground adaptation:**

- Result cards **only show included fields** — if a field isn't in the type, it isn't in the card. Reinforces the DX visually
- Raw JSON toggle shows the **actual narrowed response only** — no dual view, no full shape reference. Developer toggles includes on/off to explore
- **No TypeScript type preview panel** — type narrowing is an IDE experience, playground demonstrates runtime behavior
- Nested include toggles use **collapsible sections** per relation — parent checkbox toggles the whole relation, expand to see sub-field toggles. Matches existing ghost Button + ChevronDown collapsible pattern. Handles DigitalAsset's 17 sub-fields cleanly

## Summary

This phase adds Prisma-style conditional include type inference to the SDK's 5 domains (Profiles, Digital Assets, NFTs, Owned Assets, Owned Tokens). Currently the `include` parameter controls which GraphQL fields are fetched, but the TypeScript return type is always the full domain type with `null` for excluded fields. This phase changes that: excluded fields are **absent from the type** (not `null`), parsers **strip them at runtime**, and the type narrowing propagates through services → hooks → consumer code.

The codebase already has a clean, consistent pattern: Zod schemas define full types in `@lsp-indexer/types`, parsers in `@lsp-indexer/node` transform Hasura responses, services build include vars from the `include` parameter, and hooks in `@lsp-indexer/react` and `@lsp-indexer/next` are thin wrappers. The key change is adding a **TypeScript utility type layer** in `@lsp-indexer/types` that narrows the full type based on the include shape, then threading that generic through parsers, services, hooks, and server actions.

The approach is to define `IncludeResult<FullType, BaseFields, IncludeMap, Include>` as a mapped type utility that: (1) always includes base fields, (2) picks only the fields where `Include[key]` is `true` or where `Include` is `undefined` (full type), and (3) handles nested relation sub-includes recursively up to the existing 3-level depth. No external libraries needed — this is pure TypeScript mapped types with Zod `z.infer` providing the full shape.

**Primary recommendation:** Build a single `include-types.ts` file in `@lsp-indexer/types` with shared generic utilities, then add per-domain `GetPayload` types that map `include` parameter → narrowed return type. Thread the generic through services and hooks using the existing 1-generic-param pattern.

### Pattern 1: Include Result Mapped Type

**What:** A TypeScript utility type that narrows a full domain type based on an include parameter shape.
**When to use:** Every domain's return type.

The core type algebra:

```typescript
// packages/types/src/include-types.ts

/**
 * Base fields that are always present regardless of include parameter.
 * Every domain has `address` as the always-present field.
 */
type BaseFieldKeys = 'address';

/**
 * Map from include schema field name to the corresponding full type field name.
 * For most domains these are 1:1, but some domains have extra always-present
 * fields beyond `address` (like OwnedAsset's `id`, `digitalAssetAddress`, etc.)
 */

/**
 * Core utility: Given a full type, its base fields, and an include map,
 * produce the narrowed type.
 *
 * - `undefined` include → full type (backward compatible)
 * - `{}` include → base fields only
 * - `{ name: true }` → base fields + name
 * - `{ name: true, tags: false }` → same as `{ name: true }`
 */
type IncludeResult<
  Full,
  Base extends keyof Full,
  IncludeMap extends Record<string, keyof Full>,
  I extends Partial<Record<keyof IncludeMap, boolean | object>> | undefined,
> = I extends undefined
  ? Full // No include = full type
  : Pick<Full, Base> & {
      [K in keyof IncludeMap as K extends keyof I
        ? I[K] extends true | object
          ? IncludeMap[K]
          : never
        : never]: Full[IncludeMap[K]];
    };
```

**Key insight:** The type uses a conditional at the top level (`I extends undefined ? Full : ...`) to preserve backward compatibility. When `include` is omitted, the full type is returned. When provided, only fields explicitly set to `true` (or to a sub-include object) are included alongside base fields.

### Pattern 3: Nested Relation Include Handling

**What:** Relations like `digitalAsset`, `collection`, `holder` can be `true` (full type) or a sub-include object (narrowed).
**When to use:** NFTs, Owned Assets, Owned Tokens.

```typescript
// For a relation field, the include value can be:
// - undefined (field absent from include → field absent from result)
// - true (shorthand for full relation type)
// - { name: true, symbol: true } (narrowed relation type)

type ResolveRelation<
  Full, // e.g., DigitalAsset
  Base extends keyof Full,
  IncludeMap extends Record<string, keyof Full>,
  I, // The include value for this relation
> = I extends true
  ? Full // true = full type
  : I extends object
    ? IncludeResult<Full, Base, IncludeMap, I> // sub-include = narrowed
    : never; // absent = field not in result
```

### Pattern 4: Domain-Specific Base Fields

**What:** Some domains have always-present fields beyond `address`.
**When to use:** OwnedAsset (has `id`, `digitalAssetAddress`, `holderAddress`), OwnedToken (has `id`, `digitalAssetAddress`, `holderAddress`, `tokenId`), NFT (has `address`, `tokenId`, `isBurned`, `isMinted`).

```typescript
// OwnedAsset example — these fields are always present
type OwnedAssetBaseKeys = 'id' | 'digitalAssetAddress' | 'holderAddress';

// NFT example — address + tokenId + boolean flags always present
type NftBaseKeys = 'address' | 'tokenId' | 'isBurned' | 'isMinted';

// OwnedToken example
type OwnedTokenBaseKeys = 'id' | 'digitalAssetAddress' | 'holderAddress' | 'tokenId';
```

**Key insight from codebase analysis:** The user locked decision says "address only" as base field, but some domains have identity fields (like `id`, `tokenId`) that are always needed to identify the record. The `address` rule applies to each nesting level's concept of "identity". For top-level domains like OwnedAsset, the identity is `id` + `digitalAssetAddress` + `holderAddress`. These are the fields that the parsers will always include. The planner must clarify this with the user or infer from current schema structure.

**Actually — re-reading the context more carefully:** The user said "`address` only as the always-present base field — every domain, every nesting level." This means:

- Profile: `{ address }`
- DigitalAsset: `{ address }`
- NFT: `{ address }` (but `tokenId` is needed to identify an NFT — this may need to be a base field too)
- OwnedAsset: `{ address }` — but OwnedAsset doesn't have `address` as a field, it has `id`, `digitalAssetAddress`, `holderAddress`
- OwnedToken: same issue

**Resolution:** The context says "address" conceptually, but looking at the schema, OwnedAsset and OwnedToken don't have a top-level `address` field. They have `id` as the primary key. The planner should treat each domain's primary identifier fields as "base fields":

- Profile: `address`
- DigitalAsset: `address`
- NFT: `address` + `tokenId` (composite key)
- OwnedAsset: `id` + `digitalAssetAddress` + `holderAddress` (the identity triple)
- OwnedToken: `id` + `digitalAssetAddress` + `holderAddress` + `tokenId`

For nested relations, the base field is what the parent domain provides:

- Nested DigitalAsset: `address`
- Nested Profile (holder): `address`
- Nested NFT: `address` + `tokenId`

### Pattern 5: Parser Field Stripping

**What:** A generic utility that strips excluded fields from parsed objects.
**When to use:** Every parser, after constructing the full object.

```typescript
// packages/node/src/parsers/strip.ts

/**
 * Strip fields from a parsed object based on the include parameter.
 * - If include is undefined → return object unchanged (full type)
 * - If include is provided → delete keys not in include and not in baseFields
 */
export function stripExcluded<T extends Record<string, unknown>>(
  obj: T,
  include: Record<string, boolean | object | undefined> | undefined,
  baseFields: readonly string[],
): T {
  if (!include) return obj; // No include = return everything

  const result = { ...obj };
  for (const key of Object.keys(result)) {
    if (baseFields.includes(key)) continue; // Always keep base fields
    if (!(key in include) || include[key] === false || include[key] === undefined) {
      delete result[key];
    }
  }
  return result as T;
}
```

**Important:** This runs AFTER the existing parser logic. The parser still builds the full object, then `stripExcluded` removes fields. This approach:

1. Minimizes parser changes (just add one call at the end)
2. Handles the `false` and omission being identical rule
3. Keeps the existing `null` → removed semantic for relations

### Pattern 6: Hook Generic Propagation

**What:** TanStack Query hooks parameterized by include type so `data` is correctly narrowed.
**When to use:** All 15 hooks (5 domains × 3 patterns) across react + next packages.

```typescript
// packages/react/src/hooks/profiles.ts

export function useProfile<I extends ProfileInclude | undefined = undefined>(
  params: UseProfileParams & { include?: I },
) {
  const url = getClientUrl();
  const { address, include } = params;

  const { data, ...rest } = useQuery({
    queryKey: profileKeys.detail(address, include),
    queryFn: () => fetchProfile(url, { address, include }),
    enabled: Boolean(address),
  });

  return { profile: (data ?? null) as ProfileResult<I> | null, ...rest };
}
```

**Key insight:** The generic `I` captures the literal type of the `include` parameter at the call site. When the developer writes `useProfile({ address, include: { name: true } })`, TypeScript infers `I` as `{ name: true }` and the return type narrows to `ProfileResult<{ name: true }>` = `{ address: string; name: string | null }`.

**TanStack Query compatibility:** `useQuery` returns `UseQueryResult<TData>`. The `queryFn` return type determines `TData`. Since `fetchProfile` will be generic too, the type flows naturally. The cast `as ProfileResult<I> | null` is needed at the hook boundary because TanStack Query doesn't propagate outer generics into `queryFn`.

### Pattern 7: Derived Field Handling (standard ↔ decimals)

**What:** The `standard` field on DigitalAsset is derived from `decimals`. Per locked decision, `standard` is present only when `decimals` is included.
**When to use:** DigitalAsset include map only.

```typescript
// In the DigitalAsset include map, `decimals` maps to both `decimals` AND `standard`
// This is handled at the type level by making `standard` tied to `decimals`:

type DigitalAssetIncludeMap = {
  name: 'name';
  symbol: 'symbol';
  tokenType: 'tokenType';
  decimals: 'decimals' | 'standard'; // decimals include also brings standard
  // ... rest
};
```

**At the parser level:** When `decimals` is not in the include, `standard` is also stripped. This is already how the parser works (it derives standard from decimals presence), so the stripping just needs to also remove `standard` when `decimals` isn't included.

### Anti-Patterns to Avoid

- **Don't make the full type `null`-free:** The full `Profile` type still has `nullable` fields (like `name: string | null`). The `null` represents "data exists but value is empty." The conditional type removes the **field** entirely (it's not in the object), which is different from the field being `null`.
- **Don't create new Zod schemas per include combination:** That would be a runtime schema explosion. Zod defines the full shape only. TypeScript utility types handle narrowing.
- **Don't use `Omit` directly:** `Omit` removes known keys, but we need to _pick_ the included keys dynamically. Use `Pick` with a mapped key extraction.
- **Don't break backward compatibility:** When `include` is `undefined`, the full type MUST be returned. Every existing consumer that doesn't use `include` should continue to work without changes.

## Don't Hand-Roll

| Problem                           | Don't Build                           | Use Instead                         | Why                                                       |
| --------------------------------- | ------------------------------------- | ----------------------------------- | --------------------------------------------------------- |
| Type narrowing from include param | Custom per-domain conditional types   | Shared `IncludeResult` utility type | One utility handles all 5 domains, reduces maintenance    |
| Runtime field stripping           | Per-parser `delete` logic             | Shared `stripExcluded()` utility    | Consistent behavior, one function to test                 |
| Sub-include type resolution       | Inline conditional types per relation | `ResolveRelation` utility type      | Reusable for all nested relations across domains          |
| Include field mapping             | Hardcoded pick lists                  | `IncludeMap` type per domain        | Maps are declarative, easy to maintain when fields change |

**Key insight:** The type-level work is pure TypeScript — no runtime libraries needed. The runtime work (field stripping) is a single utility function. Resist the urge to pull in libraries for what's ~80 lines of utility types + ~20 lines of runtime stripping.

## Common Pitfalls

### Pitfall 1: TypeScript Recursion Depth Limits

**What goes wrong:** Deeply nested conditional mapped types can hit TypeScript's type instantiation depth limit (default ~50 levels).
**Why it happens:** Each level of nesting (OwnedToken → NFT → Collection → DigitalAsset fields) adds type resolution depth.
**How to avoid:** Don't build a generic recursive type. Instead, handle each nesting level explicitly. The codebase only has 3 levels max (OwnedToken → NFT → Collection[DigitalAsset]). Write the types for each level explicitly rather than using `type Recurse<T, D extends number> = ...`.
**Warning signs:** `Type instantiation is excessively deep and possibly infinite` error.

### Pitfall 2: Generic Inference Losing Literal Types

**What goes wrong:** TypeScript infers `{ name: boolean }` instead of `{ name: true }` for the include parameter, losing the conditional narrowing.
**Why it happens:** By default, TypeScript widens `true` to `boolean` in object literals unless constrained.
**How to avoid:** Use `const` type parameter or `satisfies` to preserve literal types:

```typescript
// Option A: const type parameter (TS 5.0+)
function useProfile<const I extends ProfileInclude>(params: { include?: I }) { ... }

// Option B: Narrower constraint
function useProfile<I extends ProfileInclude | undefined>(params: { include?: I }) { ... }
```

**Warning signs:** Autocomplete shows all fields even when include restricts them.

### Pitfall 3: Server Action Serialization Breaks Generics

**What goes wrong:** Next.js server actions serialize params and return values. Generic type information is lost across the serialization boundary.
**Why it happens:** Server actions use JSON serialization. TypeScript generics exist only at compile time.
**How to avoid:** The generic type parameter must be on the hook (client-side), not on the server action. The server action can return the full type, and the hook narrows it with a type assertion. Or, make the server action generic too and trust TypeScript to narrow at the call site.
**Warning signs:** Return type is always the full type in the Next.js hooks.

### Pitfall 4: `as any` Casts in Parsers Hide Type Mismatches

**What goes wrong:** The existing parsers use `as any` for nested relation parsing (e.g., `parseDigitalAsset(raw.digitalAsset as any)`). Adding generics to parsers might propagate `any` and lose type safety.
**Why it happens:** The Hasura codegen types don't perfectly match the nested sub-selections.
**How to avoid:** Keep the `as any` casts but add the generic parameter to the parser's return type, not its input type. The parser's job is to produce a correctly-shaped output — the input is already untyped due to GraphQL variance.
**Warning signs:** TypeScript errors disappear too easily — check that `as any` isn't hiding real issues.

### Pitfall 5: Breaking `null` vs Absent Semantics

**What goes wrong:** Currently `null` means "field not included." After this phase, absent means "not included" and `null` means "included but value is empty." Mixing these up breaks consumers.
**Why it happens:** The semantic change is subtle. Old code checks `if (profile.tags != null)` to detect inclusion. New code would check `if ('tags' in profile)` or rely on TypeScript preventing `profile.tags` access entirely.
**How to avoid:** Clear migration: (1) update types first, (2) update parsers to strip instead of nullify, (3) update playground cards to use `'field' in obj` checks instead of null checks. Document the semantic change.
**Warning signs:** Consumers getting runtime `undefined` when accessing stripped fields (which TypeScript correctly flags as errors).

### Pitfall 6: Empty Object `{}` Matching Everything in TypeScript

**What goes wrong:** TypeScript's structural typing means `{}` extends any object type. `include: {}` might not narrow correctly because `{}` satisfies almost any constraint.
**Why it happens:** `{}` in TypeScript is the "any non-null non-undefined" type.
**How to avoid:** The `IncludeResult` type must explicitly handle the case where no keys in `I` match the include map. When `I` is `{}`, no keys have truthy values, so only base fields should be returned. The mapped type achieves this naturally: `K extends keyof I` filters out everything when `I` is `{}`.
**Warning signs:** `include: {}` returning the full type instead of base-only.

### Pitfall 7: bigint Serialization in Next.js Server Actions

**What goes wrong:** OwnedAsset has `balance: bigint`. Server actions serialize to JSON, which can't represent `bigint`.
**Why it happens:** This is a pre-existing issue not introduced by this phase, but the runtime stripping may interact with it.
**How to avoid:** If `balance` is excluded, it gets stripped and there's no serialization issue. If included, the existing handling (likely string conversion) must continue to work.
**Warning signs:** `TypeError: Do not know how to serialize a BigInt`.

### Example 1: Complete IncludeResult Utility Type

```typescript
// packages/types/src/include-types.ts
// Confidence: HIGH — pure TypeScript, verified approach

/**
 * Extract keys from IncludeMap where the corresponding Include value is truthy.
 * Returns the full-type field names (values of IncludeMap) that should be included.
 */
type ActiveFields<IncludeMap extends Record<string, string>, I extends Record<string, unknown>> = {
  [K in keyof IncludeMap & keyof I]: I[K] extends true
    ? IncludeMap[K]
    : I[K] extends Record<string, unknown>
      ? IncludeMap[K]
      : never;
}[keyof IncludeMap & keyof I];

/**
 * Narrow a full domain type based on include parameter.
 *
 * @typeParam Full - The complete domain type (e.g., Profile)
 * @typeParam Base - Union of always-present field keys (e.g., 'address')
 * @typeParam Map - Include field name → Full type field name mapping
 * @typeParam I - The include parameter type, or undefined for full type
 */
export type IncludeResult<
  Full,
  Base extends keyof Full,
  Map extends Record<string, keyof Full & string>,
  I extends Partial<Record<keyof Map, boolean | Record<string, unknown>>> | undefined,
> = I extends undefined
  ? Full
  : Pick<Full, Base | Extract<ActiveFields<Map, NonNullable<I>>, keyof Full>>;
```

### Example 4: Runtime Field Stripping

```typescript
// packages/node/src/parsers/strip.ts

/**
 * Strip excluded fields from a parsed domain object.
 * Returns a new object containing only base fields + included fields.
 *
 * @param obj - The fully parsed object
 * @param include - The include parameter (undefined = keep everything)
 * @param baseFields - Fields to always keep
 * @param derivedFields - Map of derived fields to their source field
 *                        e.g., { standard: 'decimals' } means standard
 *                        is only kept if decimals is included
 */
export function stripExcluded<T extends Record<string, unknown>>(
  obj: T,
  include: Record<string, boolean | Record<string, unknown> | undefined> | undefined,
  baseFields: readonly string[],
  derivedFields?: Record<string, string>,
): T {
  if (!include) return obj;

  const result: Record<string, unknown> = {};

  // Always include base fields
  for (const key of baseFields) {
    if (key in obj) result[key] = obj[key];
  }

  // Include fields that are in the include parameter and truthy
  for (const [key, value] of Object.entries(include)) {
    if (value === true || (typeof value === 'object' && value !== null)) {
      if (key in obj) result[key] = obj[key];
    }
  }

  // Handle derived fields
  if (derivedFields) {
    for (const [derived, source] of Object.entries(derivedFields)) {
      if (
        include[source] === true ||
        (typeof include[source] === 'object' && include[source] !== null)
      ) {
        if (derived in obj) result[derived] = obj[derived];
      }
    }
  }

  return result as T;
}
```

## State of the Art

| Old Approach                        | Current Approach                             | When Changed  | Impact                                                                 |
| ----------------------------------- | -------------------------------------------- | ------------- | ---------------------------------------------------------------------- |
| `null` for excluded fields          | Field absent from type + stripped at runtime | This phase    | Breaking change in return type semantics — `T \| null` → field omitted |
| Full type always returned           | Narrowed type based on include               | This phase    | Better autocomplete, smaller payloads                                  |
| `T[] \| null` (null = not included) | Field absent (not in type)                   | This phase    | Simpler consumer code — no null checks for "was it included?"          |
| `const` type parameter unavailable  | `const` type parameter (TS 5.0+)             | TS 5.0 (2023) | Enables literal type inference for include objects                     |

**Deprecated/outdated:**

- The `T[] | null` where `null = field not included` semantic is being replaced. After this phase, excluded fields simply don't exist on the type.

## Open Questions

1. **OwnedAsset and OwnedToken base fields**

   - What we know: User said "`address` only" but OwnedAsset doesn't have a top-level `address` field — it has `id`, `digitalAssetAddress`, `holderAddress`
   - What's unclear: Should all identity fields be base fields, or should only `id` be base?
   - Recommendation: Treat all primary key / identity fields as base fields. For OwnedAsset: `id`, `digitalAssetAddress`, `holderAddress`. For OwnedToken: `id`, `digitalAssetAddress`, `holderAddress`, `tokenId`. Document this interpretation in the plan. The user's intent was clearly "the fields needed to identify the record are always present."

2. **Nullable base field handling for NftHolder**

   - What we know: NftHolder merges ProfileSchema with `{ timestamp }`. It's always present as `holder: NftHolderSchema.nullable()` on Nft.
   - What's unclear: When `holder` is narrowed with `{ name: true }`, should the holder type be `{ address: string; timestamp: string; name: string | null } | null` or something simpler?
   - Recommendation: The merged NftHolder type should have its own include narrowing that applies ProfileInclude to the profile fields while always keeping `timestamp` (the holder-specific field) as a base field.

3. **`const` generic parameter vs explicit literal type constraint**

   - What we know: TypeScript 5.0+ supports `const` type parameters which preserve literal types. The project uses TS 5.9.3.
   - What's unclear: Whether `const` generic parameters have any performance impact or edge cases with TanStack Query's complex types.
   - Recommendation: Use `const` type parameter as the primary approach. Fall back to explicit literal constraint if issues arise. Test with a simple PoC first.

4. **Type test strategy**
   - What we know: No existing tests in types/node/react/next packages.
   - What's unclear: How to verify type narrowing works correctly without runtime tests.
   - Recommendation: Use `@ts-expect-error` comments and `expectType`-style assertions in a `.test-d.ts` file. The `tsd` library or `vitest typecheck` can run these. Alternatively, simple `.ts` files that would fail `tsc --noEmit` if types are wrong.

## Migration Strategy Recommendation

**Order:** Types → Parsers → Services → Hooks → Playground (dependency order)

1. **Start with Profiles** (simplest domain — flat fields, no nested relations, 9 include fields)
2. **Then Digital Assets** (flat + derived field handling with standard↔decimals, 17 include fields)
3. **Then NFTs** (introduces nested relations: collection[DigitalAsset] + holder[Profile])
4. **Then Owned Assets** (nested: digitalAsset + holder)
5. **Then Owned Tokens** (most complex: 4 nested relations, deepest nesting)

Each domain should be fully migrated (types → parsers → services → react hooks → next actions/hooks → playground card) before moving to the next. This way, each step validates the pattern before increasing complexity.
