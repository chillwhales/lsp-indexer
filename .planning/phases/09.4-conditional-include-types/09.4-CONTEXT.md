# Phase 9.4: Conditional Include Types - Context

**Gathered:** 2026-02-21
**Status:** Ready for planning

<domain>
## Phase Boundary

Developer gets TypeScript return types that are narrowed based on the `include` parameter — excluded fields are absent from the type (not `null`), so autocomplete only shows what was requested. Applies to all 5 existing domains (Profiles, Digital Assets, NFTs, Owned Assets, Owned Tokens) and establishes the pattern for all future domains. Parsers are updated to strip excluded fields at runtime so the data matches the type.

</domain>

<decisions>
## Implementation Decisions

### Strictness of excluded fields

- Excluded fields are **absent from the type** — accessing them is a TypeScript error, not `undefined` or `null`
- `include: { name: true, tags: false }` and `include: { name: true }` produce the **same narrowed type** — `false` and omission are identical
- `include: {}` (empty object) returns **base fields only** — it means "I opted in to nothing." Omitting `include` entirely returns the full type
- Parsers **strip excluded fields at runtime** — `Object.keys()` only shows included fields, runtime matches the type. Not type-level-only

### Always-present base fields

- **`address` only** as the always-present base field — every domain, every nesting level. Everything else (scalars, counts, relations) is opt-in through `include`
- **Derived fields follow their source** — `standard` is present only when `decimals` is included (since it's derived from it). `tokenType` gets its own include toggle. No magic fields appearing without their source data
- **Aggregate/count fields** (`followerCount`, `holderCount`, `creatorCount`) are **regular include fields** — no special treatment, no grouping
- **Relations** follow the same rules as scalars — not listed in include = absent from type = not fetched

### Nested include depth

- `true` is shorthand for **"give me the full type"** on relations — `include: { digitalAsset: true }` returns complete DigitalAsset. Sub-include object narrows: `include: { digitalAsset: { name: true, symbol: true } }`
- Type narrowing goes **as deep as the existing Zod include schemas** — currently up to 3 levels (OwnedToken → NFT → Collection). No artificial limit, no generic recursive solution for depths that don't exist
- **Same base-field rules at every nesting level** — nested `digitalAsset` narrowed with `{ name: true }` gives `{ address, name }`. Consistent, one mental model
- **Shared sub-include types** — `DigitalAssetInclude` is reused wherever a digital asset relation appears (OwnedAsset, NFT collection, etc.). Existing exception: `OwnedTokenNftInclude` omits `collection`/`holder` from `NftInclude` — that stays

### Playground adaptation

- Result cards **only show included fields** — if a field isn't in the type, it isn't in the card. Reinforces the DX visually
- Raw JSON toggle shows the **actual narrowed response only** — no dual view, no full shape reference. Developer toggles includes on/off to explore
- **No TypeScript type preview panel** — type narrowing is an IDE experience, playground demonstrates runtime behavior
- Nested include toggles use **collapsible sections** per relation — parent checkbox toggles the whole relation, expand to see sub-field toggles. Matches existing ghost Button + ChevronDown collapsible pattern. Handles DigitalAsset's 17 sub-fields cleanly

### Claude's Discretion

- TypeScript utility type implementation strategy (mapped types, conditional types, template literal types)
- How to structure the type narrowing utilities (single file, per-domain, shared generics)
- Parser refactoring approach for stripping excluded fields
- Hook generic propagation through TanStack Query
- Migration order across the 5 existing domains
- Zod ↔ TypeScript interop approach (Zod schemas define full shape, TS utility layer narrows)

</decisions>

<specifics>
## Specific Ideas

- Prisma's `select`/`include` inference is the reference model for the DX — developer experience should feel similar
- `tokenType` mapping correction: Hasura returns `"TOKEN"` / `"NFT"` / `"COLLECTION"` strings directly — not `"0"` / `"1"` / `"2"`. Parser validates, no numeric mapping
- The existing `OwnedTokenNftIncludeSchema` (NftInclude minus collection/holder) is an established exception to shared types — preserve it

</specifics>
