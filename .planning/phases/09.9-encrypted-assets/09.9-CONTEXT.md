# Phase 9.9: Encrypted Assets - Context

**Gathered:** 2026-02-24
**Status:** Ready for planning

<domain>
## Phase Boundary

Developer can use `useEncryptedAssets` and `useInfiniteEncryptedAssets` for LSP29 encrypted asset data. Follows the established 4-plan vertical-slice pattern. Queries the `lsp29_encrypted_asset` table (rich metadata), NOT the sparse `lsp29_encrypted_asset_entry` table.

</domain>

<decisions>
## Implementation Decisions

### Domain naming

- **Type name:** `EncryptedAsset` — maps to Hasura `lsp29_encrypted_asset` table, consistent with other domain naming (DigitalAsset, OwnedAsset, IssuedAsset)
- **Hook names:** `useEncryptedAssets`, `useInfiniteEncryptedAssets` — standard plural/infinite pattern
- **No singular hook** — no `useEncryptedAsset`. Same pattern as creators. No reliable natural key exists (user-introduced elements can share address + contentId + revision)
- **Hasura table:** `lsp29_encrypted_asset` (rich metadata with title, description, images, encryption, file, chunks) — NOT `lsp29_encrypted_asset_entry` (sparse feed entries)

### Data model — base fields and include types

- **Base fields (always present):** `address`, `contentId`, `revision`
- **Includable scalar fields:** `arrayIndex`, `timestamp`
- **Includable relation fields (boolean toggles):**
  - `title` — flattened to `string | null` by parser (unwraps `{ value }` wrapper)
  - `description` — flattened to `string | null` by parser (unwraps `{ value }` wrapper)
  - `encryption` — nested include: `{ accessControlConditions?: boolean }`. Encryption scalar fields (ciphertext, dataToEncryptHash, decryptionCode, decryptionParams, method) always included when `encryption: true` or `encryption: { ... }`. Access control conditions array is separately toggled.
  - `file` — boolean, returns full file object (`hash`, `lastModified`, `name`, `size`, `type`)
  - `chunks` — boolean, returns full chunks object (`cids[]`, `iv`, `totalSize`)
  - `images` — boolean, returns image array (`imageIndex`, `url`, `width`, `height`, `verificationData`, `verificationMethod`, `verificationSource`)
  - `universalProfile` — full `ProfileInclude` sub-narrowing (same pattern as creators, issued assets, etc.)
- **Fields NOT exposed in parsed type:** `createdAt`, `rawValue`, `url`, `version`, `isDataFetched`, `retryCount`, `decodeError`, `fetchErrorCode`, `fetchErrorMessage`, `fetchErrorStatus`, `universalProfileId`, `id`

### Playground card display

- **Address lead** — address as primary identifier (consistent with all other domain cards), title/description below if included
- **Encryption details** — collapsible section ("Encryption Details" with ghost button + chevron). Shows method, ciphertext (truncated), and access control conditions as a list when expanded
- **Reusable `ImageList` component** — extract from existing duplicated pattern across ProfileCard, DigitalAssetCard, NftCard. New component handles flat array of images (thumbnail + URL + metadata). Create and backport to all existing cards in this phase.

### Playground filters

- `address` — string filter (the UP address that owns the LSP29 data, uses `_ilike`)
- `universalProfileName` — string search (nested UP → lsp3Profile → name)
- `contentId` — string filter
- `revision` — numeric filter
- `encryptionMethod` — string filter (nested `encryption.method`)
- `fileType` — string filter (nested `file.type`)
- `fileSize` — numeric filter (nested `file.size`)
- `timestamp` — date/timestamp filter

### Claude's Discretion

- Sort fields and default sort order
- Exact encryption collapsible section layout
- `ImageList` component API design (prop shape, rendering details)
- How to handle file size display (raw numeric vs formatted)
- Access control condition list rendering in encryption section

</decisions>

<specifics>
## Specific Ideas

- Reuse existing `parseImage` pattern from `parsers/utils.ts` if applicable to LSP29 images, or create LSP29-specific image parser if the shape differs (LSP29 images have `verificationData`, `verificationMethod`, `verificationSource` fields that LSP4 images don't)
- `ImageList` component should work for both LSP4 image arrays (profile avatar/profileImage/backgroundImage, digital asset icons/images) and LSP29 image arrays — accept a generic `{ url: string; [key: string]: unknown }[]` shape

</specifics>
