# Plan 09.6-03: Build Validation + Verification

**Phase:** 9.6 — Generic Type Propagation
**Requirement:** DX-05
**Packages:** All 4 packages + test app
**Depends on:** 09.6-01 (parsers + services), 09.6-02 (actions + hooks)

## Goal

Validate that all 4 packages build cleanly and the test app compiles with zero errors after the generic type propagation refactor. Verify the type narrowing works end-to-end: a consumer passing `include: { name: true }` to `useProfiles` gets back `ProfileResult<{ name: true }>` with only `address` + `name` on the type.

## Tasks

### Task 1: Build all packages

**Actions:**

1. `pnpm --filter @lsp-indexer/types build` — should succeed (no changes to types package)
2. `pnpm --filter @lsp-indexer/node build` — validates parser + service overloads compile
3. `pnpm --filter @lsp-indexer/react build` — validates hook overloads + return type aliases
4. `pnpm --filter @lsp-indexer/next build` — validates action + hook overloads
5. If any build fails, fix the issue in the relevant plan's file, then re-validate

### Task 2: Build test app

**Actions:**

1. `pnpm --filter test build` (or `next build` in apps/test)
2. Verify zero TypeScript errors — the test app imports hooks from both `@lsp-indexer/react` and `@lsp-indexer/next`, which exercises the full overload chain
3. If the test app has type errors at the page-card boundaries (e.g., `as Record<string, unknown>` casts), these are expected and should remain unchanged

### Task 3: Verify type narrowing works (spot check)

**Actions:**

1. Inspect TypeScript inference on at least one domain (profiles) — ensure:
   - `useProfiles()` returns `Profile[]` (no include → full type)
   - `useProfiles({ include: { name: true } })` returns `ProfileResult<{ name: true }>[]` (narrowed)
   - `useProfiles({ include: someVariable })` returns `PartialProfile[]` (dynamic fallback)
2. Verify `error` is typed as `Error` (not `unknown`) on hook return

### Task 4: Verify zero type assertions in updated files

**Actions:**

1. Search all updated files for `as ` type assertion patterns (excluding comments and imports)
2. Confirm no new type assertions were introduced
3. The only acceptable `as` patterns are:
   - `as Record<string, unknown>` at page-card boundaries in test app (pre-existing)
   - `as unknown as TypedDocumentString` in codegen output (auto-generated)
