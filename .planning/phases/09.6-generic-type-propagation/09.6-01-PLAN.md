# Plan 09.6-01: Parsers + Services — Generic Overloads (node package)

**Phase:** 9.6 — Generic Type Propagation
**Requirement:** DX-05
**Package:** `@lsp-indexer/node`
**Depends on:** None (first plan in phase)

## Goal

Add the 3-overload generic `<const I>` pattern to all parser and service functions across 5 domains in the `@lsp-indexer/node` package. After this plan, all `fetchX` / `fetchXs` service functions and `parseX` / `parseXs` parser functions return `XResult<I>` when a typed include is provided — matching the pattern established in followers.

## Reference Pattern

**Followers parser (gold standard):**

```typescript
// packages/node/src/parsers/followers.ts
export function parseFollower(raw: RawFollower): Follower;
export function parseFollower<const I extends FollowerInclude>(raw: RawFollower, include: I): FollowerResult<I>;
export function parseFollower(raw: RawFollower, include?: FollowerInclude): Follower | PartialFollower { ... }
```

**Followers service (gold standard):**

```typescript
// packages/node/src/services/followers.ts
export async function fetchFollows(url, params): Promise<FetchFollowsResult>;
export async function fetchFollows<const I extends FollowerInclude>(url, params: { ...; include: I }): Promise<FetchFollowsResult<FollowerResult<I>>>;
export async function fetchFollows(url, params: { ...; include?: FollowerInclude }): Promise<FetchFollowsResult<PartialFollower>>;
export async function fetchFollows(url, params): Promise<FetchFollowsResult<PartialFollower>> { ... }
```

## Notes

- Implementation bodies do NOT change — only overload declarations
- The `<const I>` is critical for TypeScript to preserve literal include types
- Generic overload must go BETWEEN the no-include and the fallback overloads (position 2 of 3)
- The `PartialX` overload stays as the catch-all for dynamic include params
