# Phase 9.6: Generic Type Propagation — Research

**Researched:** 2026-02-23
**Domain:** TypeScript overload patterns, TanStack Query typing, generic propagation
**Confidence:** HIGH

## Summary

Phase 9.5 (Social / Follows) was the first domain built from scratch with proper generic type propagation through the entire chain: parser → service → action → hook → consumer. The result is that when a consumer passes `include: { timestamp: true }` to `useFollows`, TypeScript narrows the return type to `FollowerResult<{ timestamp: true }>` — meaning only `followerAddress`, `followedAddress`, and `timestamp` exist on the returned objects, with zero type assertions anywhere in the chain.

The 5 older domains (Profiles, Digital Assets, NFTs, Owned Assets, Owned Tokens) were built before this pattern was established. They use a weaker 2-overload pattern that collapses all include-provided calls to `PartialX` — losing the precise generic narrowing. Their hooks also lack explicit return type aliases, causing `error: unknown` instead of `error: Error` in consumer code.

This phase updates all 5 older domains to match the followers pattern: 3-overload generic propagation on parsers, services, actions, and hooks — plus explicit `UseXReturn<F>` / `UseInfiniteXReturn<F>` type aliases with proper `Error` typing.

**Primary recommendation:** Replicate the followers 3-overload + return type alias pattern mechanically across all 5 older domains, then validate with `pnpm build` across all packages.

## Standard Stack

No new libraries needed. This is a pure TypeScript refactoring of existing code.

## Architecture Patterns

### Pattern: 3-Overload Generic Propagation

The followers domain established this pattern at every layer. Here's the full chain:

#### Layer 1: Parser (packages/node/src/parsers/)

**Old pattern (2 overloads):**

```typescript
export function parseProfile(raw: RawProfile): Profile;
export function parseProfile(raw: RawProfile, include: ProfileInclude): PartialProfile;
export function parseProfile(raw: RawProfile, include?: ProfileInclude): Profile | PartialProfile { ... }
```

**New pattern (3 overloads — adds generic middle overload):**

```typescript
export function parseFollower(raw: RawFollower): Follower;
export function parseFollower<const I extends FollowerInclude>(raw: RawFollower, include: I): FollowerResult<I>;
export function parseFollower(raw: RawFollower, include?: FollowerInclude): Follower | PartialFollower { ... }
```

**Key:** The `<const I>` generic preserves the literal type of the include parameter, enabling precise narrowing. The implementation signature stays the same — only the overload declarations change.

#### Layer 2: Service (packages/node/src/services/)

**Old pattern (list — 2 overloads):**

```typescript
export async function fetchProfiles(url, params?): Promise<FetchProfilesResult>;
export async function fetchProfiles(url, params: { ...; include: ProfileInclude }): Promise<FetchProfilesResult<PartialProfile>>;
export async function fetchProfiles(url, params = {}): Promise<FetchProfilesResult | FetchProfilesResult<PartialProfile>> { ... }
```

**New pattern (list — 3 overloads):**

```typescript
export async function fetchFollows(url, params): Promise<FetchFollowsResult>;
export async function fetchFollows<const I extends FollowerInclude>(url, params: { ...; include: I }): Promise<FetchFollowsResult<FollowerResult<I>>>;
export async function fetchFollows(url, params: { ...; include?: FollowerInclude }): Promise<FetchFollowsResult<PartialFollower>>;
export async function fetchFollows(url, params): Promise<FetchFollowsResult<PartialFollower>> { ... }
```

**Old detail function (2 overloads):**

```typescript
export async function fetchProfile(url, params: { address }): Promise<Profile | null>;
export async function fetchProfile(
  url,
  params: { address; include: ProfileInclude },
): Promise<PartialProfile | null>;
```

**New detail function (3 overloads):**

```typescript
export async function fetchProfile(url, params: { address }): Promise<Profile | null>;
export async function fetchProfile<const I extends ProfileInclude>(
  url,
  params: { address; include: I },
): Promise<ProfileResult<I> | null>;
export async function fetchProfile(
  url,
  params: { address; include?: ProfileInclude },
): Promise<PartialProfile | null>;
```

#### Layer 3: Server Action (packages/next/src/actions/)

**Old pattern (list — 2 overloads):**

```typescript
export async function getProfiles(params?): Promise<FetchProfilesResult>;
export async function getProfiles(params: { ...; include: ProfileInclude }): Promise<FetchProfilesResult<PartialProfile>>;
```

**New pattern (list — 3 overloads):**

```typescript
export async function getFollows(params): Promise<FetchFollowsResult>;
export async function getFollows<const I extends FollowerInclude>(params: { ...; include: I }): Promise<FetchFollowsResult<FollowerResult<I>>>;
export async function getFollows(params: { ...; include?: FollowerInclude }): Promise<FetchFollowsResult<PartialFollower>>;
```

**Old detail action (2 overloads):**

```typescript
export async function getProfile(address): Promise<Profile | null>;
export async function getProfile(address, include: ProfileInclude): Promise<PartialProfile | null>;
```

**New detail action (3 overloads):**

```typescript
export async function getProfile(address): Promise<Profile | null>;
export async function getProfile<const I extends ProfileInclude>(
  address,
  include: I,
): Promise<ProfileResult<I> | null>;
export async function getProfile(address, include?: ProfileInclude): Promise<PartialProfile | null>;
```

#### Layer 4: Hooks (packages/react/src/hooks/ AND packages/next/src/hooks/)

**Old pattern (NO overloads, loose typing):**

```typescript
export function useProfiles(params: UseProfilesParams & { include?: ProfileInclude } = {}) {
  // ...
  const profiles: PartialProfile[] = data?.profiles ?? [];
  return { profiles, totalCount: data?.totalCount ?? 0, ...rest };
}
```

**New pattern (3 overloads + return type aliases):**

```typescript
type UseFollowsReturn<F> = { follows: F[]; totalCount: number } & Omit<
  UseQueryResult<FetchFollowsResult<F>, Error>, 'data'
>;

type UseInfiniteFollowsReturn<F> = {
  follows: F[];
  hasNextPage: boolean;
  fetchNextPage: UseInfiniteQueryResult['fetchNextPage'];
  isFetchingNextPage: boolean;
} & Omit<
  UseInfiniteQueryResult<InfiniteData<FetchFollowsResult<F>>, Error>,
  'data' | 'hasNextPage' | 'fetchNextPage' | 'isFetchingNextPage'
>;

export function useFollows<const I extends FollowerInclude>(
  params: UseFollowsParams & { include: I },
): UseFollowsReturn<FollowerResult<I>>;
export function useFollows(
  params: Omit<UseFollowsParams, 'include'> & { include?: never },
): UseFollowsReturn<Follower>;
export function useFollows(
  params: UseFollowsParams & { include?: FollowerInclude },
): UseFollowsReturn<PartialFollower>;
export function useFollows(
  params: UseFollowsParams & { include?: FollowerInclude },
): UseFollowsReturn<PartialFollower> { ... }
```

**Key differences in hooks:**

1. **3 overloads** with generic first (for IDE autocomplete on include)
2. **Explicit return type aliases** with `UseQueryResult<T, Error>` — fixes `error: unknown` → `error: Error`
3. **`InfiniteData<>` wrapper** on infinite hook return types
4. **`Omit<UseFollowsParams, 'include'> & { include?: never }`** on the no-include overload

### Anti-Patterns to Avoid

- **Don't change implementation bodies** — only overload signatures and return type annotations change. The runtime code is identical.
- **Don't forget `const` on the generic** — `<I extends XInclude>` won't preserve literal types; must be `<const I extends XInclude>`
- **Don't reorder overloads** — generic must be FIRST for IDE autocomplete to work (TypeScript uses first overload for contextual typing)

## Don't Hand-Roll

| Problem               | Don't Build                    | Use Instead                                    | Why                                                             |
| --------------------- | ------------------------------ | ---------------------------------------------- | --------------------------------------------------------------- |
| Narrowed return types | Custom mapped types per domain | Existing `XResult<I>` types from types package | Already built in Phase 9.4, tested across all domains           |
| Explicit error typing | Custom error type wrappers     | `UseQueryResult<T, Error>` from TanStack Query | Standard TanStack pattern, fixes `error: unknown` automatically |
| Infinite data typing  | Custom page aggregation types  | `InfiniteData<T>` from TanStack Query          | Built-in TanStack type for infinite query data shape            |

## Common Pitfalls

### Pitfall 1: Overload Order Matters for IDE Autocomplete

**What goes wrong:** If no-include overload is first, IDE sees `include?: never` and offers no key suggestions inside `include: { ... }`
**Why it happens:** TypeScript's language service uses the first overload for contextual typing
**How to avoid:** Generic overload (`<const I>`) must always be the FIRST overload
**Warning signs:** IDE doesn't autocomplete include fields

### Pitfall 2: Implementation Signature Return Type Must Be Widest

**What goes wrong:** TypeScript error if implementation return type doesn't cover all overload return types
**Why it happens:** Implementation must be compatible with ALL overloads
**How to avoid:** Implementation signature always returns `Partial*` (the widest type), e.g., `UseFollowsReturn<PartialFollower>`

### Pitfall 3: Detail Functions Have Different Overload Shape

**What goes wrong:** Detail functions (`fetchProfile`, `getProfile`, `useProfile`) return `T | null`, not `FetchResult<T>`. Different overload shape than list functions.
**Why it happens:** Detail functions return a single item or null, not a paginated result
**How to avoid:** Keep `XResult<I> | null` for detail, `FetchXsResult<XResult<I>>` for list

### Pitfall 4: NFT Detail Has 3 Positional Params in Action

**What goes wrong:** `getNft(address, tokenId, formattedTokenId, include)` — 4 positional params, not object-based
**Why it happens:** Historical API design, can't change without breaking consumers
**How to avoid:** Add generic overload maintaining same positional param pattern

### Pitfall 5: Profile Detail Action Uses Positional Params

**What goes wrong:** `getProfile(address, include)` — 2 positional params (not object-based like list actions)
**Why it happens:** Same historical pattern
**How to avoid:** Maintain positional shape, just add generic overload between existing overloads

### Return Type Alias Pattern Per Domain

Each domain needs 3 type aliases (for list hooks):

```typescript
// Standard hook return (detail)
type UseXReturn<F> = { x: F | null } & Omit<UseQueryResult<F | null, Error>, 'data'>;

// Standard hook return (list)
type UseXsReturn<F> = { xs: F[]; totalCount: number } & Omit<
  UseQueryResult<FetchXsResult<F>, Error>,
  'data'
>;

// Infinite hook return
type UseInfiniteXsReturn<F> = {
  xs: F[];
  hasNextPage: boolean;
  fetchNextPage: UseInfiniteQueryResult['fetchNextPage'];
  isFetchingNextPage: boolean;
} & Omit<
  UseInfiniteQueryResult<InfiniteData<FetchXsResult<F>>, Error>,
  'data' | 'hasNextPage' | 'fetchNextPage' | 'isFetchingNextPage'
>;
```
