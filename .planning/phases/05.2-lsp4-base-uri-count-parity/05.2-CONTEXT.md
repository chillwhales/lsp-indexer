<decisions>
## Implementation Decisions

### LSP4 Metadata Base URI Derivation (GAP-06)

- LSP8 collections can have LSP4 metadata with base URI via `DataChanged` (LSP8TokenMetadataBaseURI key) and per-token LSP4 metadata via `TokenIdDataChanged`
- Two trigger paths must be implemented:
  1. **New NFT discovered (mint):** When an NFT is minted (LSP8 Transfer from zero address), check if the parent LSP8 collection has an existing `LSP8TokenMetadataBaseURI`. If yes, derive the per-token URL and create/fetch an `LSP4Metadata` entity. If no base URI exists, move on.
  2. **Base URI updated:** When `LSP8TokenMetadataBaseURI` changes via `DataChanged`, update/create `LSP4Metadata` entities for ALL existing NFTs belonging to that LSP8 collection.
- URL derivation format matches V1: `baseUri.endsWith('/') ? baseUri + formattedTokenId : baseUri + '/' + formattedTokenId`
- Entity ID format: `"BaseURI - {address} - {tokenId}"` (matching V1)
- This is the big gap (~84K missing entities out of 116K total LSP4Metadata in V1)

### OwnedAsset Double-Processing Fix (GAP-08)

- Root cause: `ownedAssets.handler.ts` ignores the `triggeredBy` parameter and processes ALL transfers from BOTH bags on every invocation. Since `listensToBag: ['LSP7Transfer', 'LSP8Transfer']`, the handler is called twice per batch, doubling balances.
- Fix: Use the `triggeredBy` parameter to only process that bag's transfers (matching how `nft.handler.ts` and other dual-trigger handlers already work)
- This will eliminate the ~14K inflated entities where doubled balances prevented legitimate zero-balance deletions

### Orb Entity Mint Defaults (GAP-09)

- Root cause: V2's orb handlers (`orbLevel.handler.ts`, `orbFaction.handler.ts`) only listen to `TokenIdDataChanged` events. V1 also creates default entities on mint (Transfer from zero address to ORBS_ADDRESS).
- Fix: Add mint detection to orb handlers — listen to `LSP8Transfer` in addition to `TokenIdDataChanged`
- On mint of an Orb NFT, create defaults: `OrbLevel=0`, `OrbCooldownExpiry=0`, `OrbFaction="Neutral"`
- On subsequent `TokenIdDataChanged` events, overwrite defaults with actual on-chain values (existing behavior)

### LSP8ReferenceContract — No Fix Needed (GAP-07)

- V1 has a switch fall-through bug in `scanner.ts` (lines 203-228): missing `break` statements cause `LSP4Creators[].length`, `LSP5ReceivedAssets[].length`, and `AddressPermissions[].length` DataChanged events to fall through and create phantom `LSP8ReferenceContract` entities
- V2 correctly filters by exact data key — V2's count is the correct one
- Mark as known V1 divergence in comparison tool's entity registry (V1 count > V2 count, reason: V1 fall-through bug)
