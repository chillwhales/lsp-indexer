# Phase 20.1: Structured Logging Overhaul - Context

**Gathered:** March 14, 2026
**Status:** Ready for planning

<domain>
## Phase Boundary

Migrate all 40 unstructured log calls across 8 indexer files to queryable `(attrs, message)` structured format for Grafana/Loki. This enables field-based queries and monitoring dashboards. Scope includes console.*, JSON.stringify(), and template string patterns.

</domain>

<decisions>
## Implementation Decisions

### Field naming schema
- Convention with core fields: Every log call MUST include `step` (EXTRACT, HANDLE, BOOTSTRAP, etc.) and `handler`/`plugin` (component name)
- Standardize common optional fields with recommended vocabulary (`address`, `category`, `durationMs`, `entityType`, `count`, etc.)
- Create reference document (`LOGGING.md`) listing core fields + common optional fields with examples

### Registry logger injection
- Constructor injection: Pass logger to `PluginRegistry` and `EntityHandlerRegistry` constructors
- Use same schema: Registry logs use `step: 'BOOTSTRAP'`, `component: 'registry'` for consistency
- Current detail level: Migrate existing log content with structured fields, preserve debugging information
- Registry-only logging: Only registry logs during registration, plugins/handlers stay silent until runtime

### Worker thread strategy
- Post messages to parent: Send log data as messages to main thread for structured logging in Grafana
- Dedicated log message type: Add `LOG` message type alongside existing worker messages
- Same schema with worker context: Use `step: 'METADATA_FETCH'`, `component: 'worker'` plus `workerId`, `threadId`
- Migrate all worker logs: Send all console.* calls (error, warn, info) as messages for complete migration

### Log level audit
- Migrate as-is: Keep existing log levels (console.warn → log.warn, console.error → log.error)
- Keep existing variety: Each file's levels reflect author's context understanding, don't standardize
- Migration only: Only migrate existing console.* calls, don't add new log statements
- Verify during testing: Include log level verification in testing plan to ensure correct Grafana behavior

### Claude's Discretion
- Exact field vocabulary terms (as long as they follow the core + common pattern)
- Implementation details of worker message handling
- LOGGING.md document structure and examples

</decisions>
