Implement BORD-04: UP, DA, and NFT entities retain their first-seen block position. Later updates never overwrite these fields.

**Step 1: Update verification.ts — core entity creation with block fields**

Modify the `verifyWithInterface()` function signature to accept a block position map. When creating new UP/DA entities (lines ~355-363), pass block fields:

```typescript
// Before (current):
newEntities.set(addr, new UniversalProfile({ id: addr, address: addr }));

// After:
const blockPos = blockPositionByAddress.get(addr);
newEntities.set(addr, new UniversalProfile({
  id: addr,
  address: addr,
  blockNumber: blockPos?.blockNumber ?? 0,
  transactionIndex: blockPos?.transactionIndex ?? 0,
  logIndex: blockPos?.logIndex ?? 0,
}));
```

Update the `createVerifyFn()` returned function to accept an additional parameter for the block position map. The map type:

```typescript
type BlockPosition = { blockNumber: number; transactionIndex: number; logIndex: number };
```

Update the `VerifyFn` type in `pipeline.ts` to pass this data through.

**Step 2: Update pipeline.ts — compute earliest enrichment per address**

In the pipeline's Step 5 section (after the enrichment queue grouping loop, before calling `verifyAddresses`), compute the earliest block position for each address:

```typescript
// Compute earliest block position per address for BORD-04
const earliestBlockByAddress = new Map<string, BlockPosition>();
for (const request of enrichmentQueue) {
  if (request.category === EntityCategory.NFT) continue;
  const existing = earliestBlockByAddress.get(request.address);
  if (!existing || compareBlockPosition(request, existing) < 0) {
    earliestBlockByAddress.set(request.address, {
      blockNumber: request.blockNumber,
      transactionIndex: request.transactionIndex,
      logIndex: request.logIndex,
    });
  }
}
```

Add a helper function `compareBlockPosition(a, b)` that compares by blockNumber first, then transactionIndex, then logIndex. Return negative if a is earlier.

Pass `earliestBlockByAddress` to the verify function call.

**Step 3: Ensure block fields are NOT overwritten on existing core entities**

The current code already handles this correctly: `verifyWithInterface()` only creates entity instances for NEWLY verified addresses (not seen before). Existing addresses return from cache/DB without new entity creation. So existing UP/DA entities' block fields are naturally preserved.

However, verify this is also true for the pipeline's `createFkStub()` function — FK stubs (used in Step 6 ENRICH) are minimal objects with only `id`. They should NOT include block fields, ensuring upserts in Step 6 don't overwrite them. Current code already creates stubs with just `{ id: ... }` — confirm this remains unchanged.

**Step 4: Handle NFT block position**

NFT entities are created by the `nft.handler.ts` (Plan 03's scope), not by verification. However, NFT enrichment requests also need the earliest block position. The NFT handler already has access to Transfer events which carry block fields — Plan 03 will handle setting block fields on NFTs.

For the pipeline, ensure that NFT FK stubs created in `createFkStub()` do NOT include block fields (they're just ID references).
