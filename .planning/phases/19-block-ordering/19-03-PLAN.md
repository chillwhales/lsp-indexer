Wire block ordering data through all EntityHandlers for derived entities.

This plan updates all ~29 handlers to (a) set block fields on derived entity constructors from the triggering event, and (b) pass real block values in enrichment requests. Metadata fetch handlers propagate block fields from parent metadata entities to their sub-entities.

For each handler, apply the appropriate pattern based on its category:

**Category A: Data-key handlers (listen to DataChanged)**
These handlers read `DataChanged` events which already have `blockNumber`, `transactionIndex`, `logIndex`. Copy them to derived entity constructors and enrichment requests.

Handlers: `lsp4TokenName`, `lsp4TokenSymbol`, `lsp4TokenType`, `lsp4Creators` (LSP4CreatorsLength portion), `lsp5ReceivedAssets` (LSP5ReceivedAssetsLength + LSP5ReceivedAsset), `lsp6Controllers`, `lsp8TokenIdFormat`, `lsp8ReferenceContract`, `lsp8MetadataBaseURI`, `lsp12IssuedAssets`, `lsp29EncryptedAsset`, `lsp3Profile`, `lsp4Metadata`.

**IMPORTANT — lsp4MetadataBaseUri is NOT a Category A handler.** Despite the similar name to `lsp8MetadataBaseURI` (which IS Category A), `lsp4MetadataBaseUri` listens to `['LSP8Transfer', 'LSP8TokenMetadataBaseURI']`, NOT `DataChanged`. It has two trigger paths:
1. **LSP8Transfer** (mint) — block fields come from the Transfer entity: `event.blockNumber`, `event.transactionIndex`, `event.logIndex`
2. **LSP8TokenMetadataBaseURI** (base URI change, from TokenIdDataChanged bag) — block fields come from the TokenIdDataChanged entity

Use the respective triggering event's block fields for entity constructors and enrichment requests. Apply the same approach as Category B transfer-derived handlers, with an additional branch for TokenIdDataChanged triggers.

Pattern for entity constructor (example — lsp4TokenName):
```typescript
const entity = new LSP4TokenName({
  id: event.address,
  address: event.address,
  timestamp: event.timestamp,
  blockNumber: event.blockNumber,           // NEW
  transactionIndex: event.transactionIndex, // NEW
  logIndex: event.logIndex,                 // NEW
  value: ...,
  rawValue: event.dataValue,
  digitalAsset: null,
});
```

Pattern for enrichment request:
```typescript
hctx.batchCtx.queueEnrichment<LSP4TokenName>({
  category: EntityCategory.DigitalAsset,
  address: event.address,
  entityType: ENTITY_TYPE,
  entityId: entity.id,
  fkField: 'digitalAsset',
  blockNumber: event.blockNumber,           // NEW (replaces 0)
  transactionIndex: event.transactionIndex, // NEW (replaces 0)
  logIndex: event.logIndex,                 // NEW (replaces 0)
});
```

**IMPORTANT for lsp5ReceivedAssets**: This handler has complex merge logic (Index + Map events). When creating new entities, use the current event's block fields. When merging into existing entities (the `...existing` spread pattern), preserve the EXISTING entity's block fields (don't overwrite with the second event's block fields):
```typescript
const updated = new LSP5ReceivedAsset({
  ...existing,
  arrayIndex: existing.arrayIndex ?? arrayIndex,
  timestamp,
  // Do NOT add blockNumber/transactionIndex/logIndex here — keep existing values from first creation
});
```
But for NEW entities, set them from the event.

**IMPORTANT for lsp6Controllers**: This handler creates multiple sub-entity types (LSP6Controller, LSP6Permission, LSP6AllowedCall, LSP6AllowedERC725YDataKey). ALL sub-entities get block fields from the DataChanged event.

**IMPORTANT for lsp4Creators**: Creates both LSP4CreatorsLength AND LSP4Creator entities. Both get block fields from the DataChanged event.

**Category B: Transfer-derived handlers**
These read Transfer events (LSP7Transfer/LSP8Transfer) which have block fields.

- `nft.handler.ts`: NFT entities get block fields from the Transfer event (mint/burn). For TokenIdDataChanged-triggered stubs, use the TokenIdDataChanged event's block fields.
- `totalSupply.handler.ts`: TotalSupply entities get block fields from the Transfer. When accumulating across multiple transfers, use the LATEST transfer's block fields (since TotalSupply tracks current value, the most recent block is appropriate).
- `ownedAssets.handler.ts`: OwnedAsset/OwnedToken already use `blockNumber` (renamed from `block` in Plan 01). Add `transactionIndex: transfer.transactionIndex` and `logIndex: transfer.logIndex` to all entity constructors and `...existing` spreads. The Transfer entity already has these fields.
- `follower.handler.ts`: Follower entities get block fields from Follow/Unfollow events.

**Category C: Post-verification handlers**
These run after Step 5 and read from verified entities.

- `decimals.handler.ts`: Creates Decimals from newly verified DAs. Get block fields from the DA entity in `newEntities`. Cast `da` to access block fields: `(da as { id: string; blockNumber?: number; transactionIndex?: number; logIndex?: number })`. If block fields aren't available (shouldn't happen after Plan 02), default to 0.
- `universalProfileOwner.handler.ts`: Creates owner entities. Get block fields from the UP entity in `newEntities`, same casting approach.
- `digitalAssetOwner.handler.ts`: Creates owner entities. Get block fields from the DA entity, same pattern.

**Category D: formattedTokenId handler**
- `formattedTokenId.handler.ts`: This handler reads existing NFT entities and updates their `formattedTokenId` field. It creates new NFT instances with `...existing` spread — the block fields will already be on the existing entity. No new block field logic needed, just ensure the spread preserves them. Verify enrichment requests use the NFT's existing block fields.

// Example for LSP3ProfileName

  id: uuidv4(),
