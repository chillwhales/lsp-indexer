2. **Create `handlers/totalSupply.handler.ts`:**

   File-level JSDoc:

   ```
   /**
    * TotalSupply entity handler.
    *
    * Subscribes to LSP7Transfer and LSP8Transfer events and maintains running
    * total supply per digital asset contract. Filters for mint (from === zeroAddress)
    * and burn (to === zeroAddress) transfers, accumulates deltas in-memory across
    * all transfers in the batch, and writes final TotalSupply entities to BatchContext.
    *
    * Behavior:
    *   - Reads existing TotalSupply from DB + own BatchContext derived key (for dual-trigger accumulation)
    *   - Mints increment, burns decrement
    *   - Clamps to zero on underflow (burn > supply) with console.warn
    *   - Queues enrichment for digitalAsset FK
    *
    * Port from v1 via: core/handlerHelpers.ts:updateTotalSupply() (dead code)
    */
   ```

   Implementation:

   ```typescript
   const ENTITY_TYPE = 'TotalSupply';

   const TotalSupplyHandler: EntityHandler = {
     name: 'totalSupply',
     listensToBag: ['LSP7Transfer', 'LSP8Transfer'],

     async handle(hctx: HandlerContext, triggeredBy: string): Promise<void> {
       const transfers = [...hctx.batchCtx.getEntities<Transfer>(triggeredBy).values()];

       // Filter for mint/burn only
       const mintBurnTransfers = transfers.filter(
         ({ from, to }) =>
           isAddressEqual(zeroAddress, getAddress(from)) ||
           isAddressEqual(zeroAddress, getAddress(to)),
       );
       if (mintBurnTransfers.length === 0) return;

       // Collect unique contract addresses
       const addresses = [...new Set(mintBurnTransfers.map((t) => t.address))];

       // Load existing state from BOTH DB and BatchContext (dual-trigger accumulation)
       // When handle() is called for LSP7Transfer first, those results go into BatchContext.
       // When called again for LSP8Transfer, we must read the first call's results from BatchContext.
       const dbEntities = await hctx.store.findBy(TotalSupply, { id: In(addresses) });
       const dbMap = new Map(dbEntities.map((e) => [e.id, e]));
       const batchEntities = hctx.batchCtx.getEntities<TotalSupply>(ENTITY_TYPE);

       // Merge: BatchContext takes precedence over DB (more recent)
       const currentState = new Map<string, TotalSupply>();
       for (const addr of addresses) {
         const fromBatch = batchEntities.get(addr);
         const fromDb = dbMap.get(addr);
         if (fromBatch) currentState.set(addr, fromBatch);
         else if (fromDb) currentState.set(addr, fromDb);
       }

       for (const transfer of mintBurnTransfers) {
         const { timestamp, address, from, to, amount } = transfer;

         let entity = currentState.get(address);
         if (!entity) {
           entity = new TotalSupply({
             id: address,
             timestamp,
             address,
             digitalAsset: null, // FK initially null — resolved by enrichment
             value: 0n,
           });
         }

         // Mint: from === zeroAddress → increment
         if (isAddressEqual(zeroAddress, getAddress(from))) {
           entity = new TotalSupply({
             ...entity,
             timestamp,
             value: entity.value + amount,
           });
         }

         // Burn: to === zeroAddress → decrement (clamp to zero)
         if (isAddressEqual(zeroAddress, getAddress(to))) {
           if (entity.value < amount) {
             console.warn(
               `[totalSupply] Underflow for ${address}: supply ${entity.value} < burn ${amount}. Clamping to 0.`,
             );
           }
           entity = new TotalSupply({
             ...entity,
             timestamp,
             value: entity.value > amount ? entity.value - amount : 0n,
           });
         }

         currentState.set(address, entity);
       }

       // Write all updated entities to BatchContext + queue enrichment
       for (const [id, entity] of currentState) {
         hctx.batchCtx.addEntity(ENTITY_TYPE, id, entity);
         hctx.batchCtx.queueEnrichment<TotalSupply>({
           category: EntityCategory.DigitalAsset,
           address: id,
           entityType: ENTITY_TYPE,
           entityId: id,
           fkField: 'digitalAsset',
         });
       }
     },
   };

   export default TotalSupplyHandler;
   ```

File-level JSDoc:

```
/**
 * OwnedAssets entity handler.
 *
 * Subscribes to LSP7Transfer and LSP8Transfer events and maintains OwnedAsset
 * and OwnedToken records tracking which addresses hold which digital assets.
 *
 * Behavior:
 *   - Reads existing OwnedAsset/OwnedToken state from DB + own BatchContext keys
 *   - Sender: decrements balance (floor at 0), deletes OwnedAsset if balance=0
 *   - Receiver: increments balance, creates new OwnedAsset if first-time holder
 *   - For LSP8 (NFTs): deletes sender's OwnedToken, creates receiver's OwnedToken
 *   - Queues deletions via queueDelete() — OwnedTokens deleted BEFORE OwnedAssets (FK ordering)
 *   - Queues enrichment for digitalAsset + universalProfile FKs
 *   - OwnedToken also queues enrichment for nft FK
 *
 * Port from v1 via: core/handlerHelpers.ts:updateOwnedAssets() (dead code)
 */
```

Key implementation details:

1. **Handler structure:**

   ```typescript
   const OWNED_ASSET_TYPE = 'OwnedAsset';
   const OWNED_TOKEN_TYPE = 'OwnedToken';

   const OwnedAssetsHandler: EntityHandler = {
     name: 'ownedAssets',
     listensToBag: ['LSP7Transfer', 'LSP8Transfer'],
     async handle(hctx, triggeredBy): Promise<void> { ... },
   };
   export default OwnedAssetsHandler;
   ```

2. **Dual-trigger accumulation** (same pattern as totalSupply):

   - Read existing state from BOTH DB AND BatchContext for both OwnedAsset and OwnedToken
   - BatchContext entities from a prior trigger call take precedence over DB

3. **Processing transfers** (port from handlerHelpers.ts lines 194-307):

   - For each transfer, compute OwnedAsset IDs using `generateOwnedAssetId()` (new colon format)
   - Sender decrement: find existing, reduce balance, floor at 0
   - Receiver increment: find existing or create new, increase balance
   - For NFTs (when `transfer.tokenId` is truthy):
     - Sender: mark for deletion (set tokenId to null on OwnedToken)
     - Receiver: create new OwnedToken or restore existing
     - Use `generateOwnedTokenId()` (new colon format)
   - OwnedToken.ownedAsset FK: set to `new OwnedAsset({ id: parentAssetId })` if parent exists in batch or DB

4. **Deletion via queueDelete():**
   After processing all transfers, separate entities into save vs delete:

   ```typescript
   const ownedAssetsToDelete = [...updatedOwnedAssets.values()].filter((e) => e.balance === 0n);
   const ownedTokensToDelete = [...updatedOwnedTokens.values()].filter((e) => !e.tokenId);
   ```

   Queue deletions in correct FK order (OwnedTokens FIRST, then OwnedAssets):

   ```typescript
   if (ownedTokensToDelete.length > 0) {
     hctx.batchCtx.queueDelete({ entityClass: OwnedToken, entities: ownedTokensToDelete });
   }
   if (ownedAssetsToDelete.length > 0) {
     hctx.batchCtx.queueDelete({ entityClass: OwnedAsset, entities: ownedAssetsToDelete });
   }
   ```

5. **Write survivors to BatchContext + queue enrichment:**
   For OwnedAssets with balance > 0:
