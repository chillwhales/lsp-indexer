  <files>packages/indexer-v2/src/handlers/chillwhales/chillClaimed.handler.ts</files>
  <action>
Create an async EntityHandler that ports V1's `chillClaimedHandler.ts` + `utils/chillClaimed/index.ts` to V2's EntityHandler interface.

**Handler structure:**

```typescript
import { CHILL_ADDRESS, CHILLWHALES_ADDRESS } from '@/constants/chillwhales';
import { EntityCategory, EntityHandler, HandlerContext } from '@/core/types';
import { aggregate3StaticLatest } from '@/core/multicall';
import { generateTokenId, isNullAddress } from '@/utils';
import { CHILL } from '@chillwhales/abi';
import { Aggregate3StaticReturn } from '@chillwhales/abi/lib/abi/Multicall3';
import { ChillClaimed, NFT, Transfer } from '@chillwhales/typeorm';
import { hexToBool, isHex } from 'viem';
```

- `name: 'chillClaimed'`
- `listensToBag: ['LSP8Transfer']`
- `postVerification: false` (no postVerification needed — this is a Step 3 handler)

**Phase 1 — Mint detection (runs every batch):**

1. Get all LSP8Transfer entities via `hctx.batchCtx.getEntities<Transfer>('LSP8Transfer')`
2. Filter to: `event.address.toLowerCase() === CHILLWHALES_ADDRESS.toLowerCase()` AND `isNullAddress(event.from)` (mint = transfer from zero address)
3. For each matching mint transfer, create a `ChillClaimed` entity:
   - `id: generateTokenId({ address: CHILLWHALES_ADDRESS, tokenId: event.tokenId })`
   - `address: CHILLWHALES_ADDRESS`
   - `digitalAsset: null` (FK resolved via enrichment queue)
   - `tokenId: event.tokenId`
   - `nft: null` (FK resolved via enrichment queue)
   - `value: false` (not yet verified on-chain)
4. Add entity via `batchCtx.addEntity('ChillClaimed', id, entity)`
5. Queue enrichment for `digitalAsset` FK (category: DigitalAsset, address: CHILLWHALES_ADDRESS)
6. Queue enrichment for `nft` FK (category: NFT, address: CHILLWHALES_ADDRESS, tokenId: event.tokenId)

**Phase 2 — On-chain verification (runs at chain head only):**

1. Check `hctx.isHead` — if false, skip this phase entirely
2. Query the database for unclaimed ChillClaimed entities: `hctx.store.findBy(ChillClaimed, { value: false })`
   - Note: V1 uses `value: true` in findBy then excludes those from multicall. But the semantics are: find NFTs that haven't been verified yet. In V1, entities start with `value: false` and get updated to `value: true` when verified. The findBy should get entities where `value` is `false` (unclaimed ones that need checking).
   - ACTUALLY, re-reading V1 carefully: V1 queries `findBy(ChillClaimed, { value: true })` to get ALREADY CLAIMED entities, then queries NFTs NOT IN those tokenIds. This is a different approach — V1 finds all NFTs without a `ChillClaimed(value=true)` record.
   - For V2, simplify: query `hctx.store.findBy(ChillClaimed, { value: false })` to get all entities inserted with value=false that haven't been verified yet. This is equivalent because we only insert with value=false.
3. If no unclaimed entities exist, skip
4. Batch call `CHILL.functions.getClaimedStatusFor.encode({ tokenId })` via `aggregate3StaticLatest` with batch size 500 (matching V1)
5. For results where `success && isHex(returnData) && hexToBool(returnData)`:
   - Update entity value to `true` and upsert via batchCtx
6. Add updated entities via `batchCtx.addEntity('ChillClaimed', id, updatedEntity)` — pipeline upserts handle the update

**Important V2 adaptations:**

- Use `hctx.context` (not `context`) for RPC access in `aggregate3StaticLatest`
- The V2 multicall function signature is `aggregate3StaticLatest(context, calls)` — note it takes context as first arg, calls as second (not the V1 `{ context, calls }` object pattern)
- Add a 1-second delay between batches via `await new Promise(resolve => setTimeout(resolve, 1000))` to match V1's rate limiting
- Use structured logging via `hctx.context.log.info(JSON.stringify({...}))` for progress


- `name: 'orbsClaimed'`

1. Entity type: `OrbsClaimed` instead of `ChillClaimed`
2. Multicall: `ORBS.functions.getChillwhaleClaimStatus` on `ORBS_ADDRESS` instead of `CHILL.functions.getClaimedStatusFor` on `CHILL_ADDRESS`

Everything else — mint detection filter (CHILLWHALES_ADDRESS), entity ID generation, enrichment queuing, batch pattern — is identical.
