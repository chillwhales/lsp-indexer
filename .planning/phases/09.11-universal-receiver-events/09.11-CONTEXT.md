# Phase 9.11: Universal Receiver Events - Context

**Gathered:** 2026-02-25
**Status:** Ready for planning

<domain>
## Phase Boundary

Developer can use `useUniversalReceiverEvents`, `useInfiniteUniversalReceiverEvents` for universal receiver event history — LSP1 events emitted when a Universal Profile receives tokens, vaults, or other notifications. Single domain from the Hasura `universal_receiver` table.

</domain>

<decisions>
## Implementation Decisions

### typeId Resolution

- Build a new `@lsp-indexer/lsp1-type-ids` standalone package — separate from data-keys, mirrors the same pattern (Zod enum schema, resolver function, zero framework deps)
- Hardcode hex values with source comments — zero `@lukso/*` contract dependencies (same lesson as data-keys)
- Parser derives `typeIdName` field from `typeId` (like `dataKeyName` from `dataKey` in data-changed-events)
- `typeIdName` is a filterable field — Zod enum schema (`Lsp1TypeIdNameSchema`) enables type-safe filtering with IDE autocomplete, service layer resolves to hex

### Hook Variants

- No `useLatestUniversalReceiverEvent` hook — developers use `useUniversalReceiverEvents` with `sort: { field: 'timestamp', direction: 'desc' }` + `limit: 1` if needed
- No singular `useUniversalReceiverEvent` hook — no natural key exists (opaque Hasura ID), matching creators/issued-assets/encrypted-assets pattern
- Two hooks only: `useUniversalReceiverEvents` (paginated list) + `useInfiniteUniversalReceiverEvents` (infinite scroll)
- `value` exposed as BigInt via `z.bigint()` — Hasura numeric → BigInt conversion, matching `OwnedAsset.balance` pattern
- `receivedData` and `returnedValue` are both includable fields (not base) — raw hex blobs, most consumers won't need them until decoding exists

### Playground Card

- typeIdName as card header — bold resolved name when known, "(Unknown Type)" in muted italic when null, raw hex truncated below (matching dataKeyName display pattern from data-changed-events)
- Single "From" section with dynamic content — renders either a profile block or digital asset block depending on which relation exists, with a "Profile" / "Digital Asset" badge indicating entity type. Uses `'key' in obj` guard pattern.
- Value displayed same as balance/totalSupply in existing cards — consistent formatting via `formatTokenAmount` helpers regardless of zero/non-zero
- 2-tab playground layout: List + Infinite (no Latest tab since no latest hook, no Single tab since no singular hook)

### Claude's Discretion

- Base fields selection (which fields are always present vs includable beyond the ones discussed)
- Filter fields and sort fields (beyond typeIdName filter which is locked)
- Include schema structure (scalar toggles + relation sub-includes)
- GraphQL document variable naming and prefix conventions
- Exact LSP1 type IDs to include in the registry (research the LUKSO specs)

</decisions>

<specifics>
## Specific Ideas

- typeId resolution mirrors data-changed-events' dataKeyName resolution exactly — same DX pattern, same resolver approach, separate package
- Dual "from" relations (fromProfile + fromAsset) are unique to this domain — card handles this with a single adaptive section
- Value is BigInt (wei) using same formatTokenAmount helpers already in the test app

</specifics>

<deferred>
## Deferred Ideas

- **ABI decoding of receivedData based on typeId** — decode raw hex payloads into structured fields per event type (e.g., LSP7 transfer → { sender, receiver, amount }). Requires ABI definitions per type ID, a decoding library (viem/ethers), and typed return structures. Significant new capability — its own phase.

</deferred>
