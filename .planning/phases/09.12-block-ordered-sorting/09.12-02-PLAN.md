```graphql

$includeTransactionIndex: Boolean! = true
$includeLogIndex: Boolean! = true
```

Add 3 new field selections inside the `follower(...)` selection set, after the `address @include(if: $includeAddress)` line:

```graphql

log_index @include(if: $includeLogIndex)
```

These follow the exact same pattern as `timestamp @include(if: $includeTimestamp)`. The Hasura `Follower_Order_By` codegen type already has these fields — they exist in the schema, just weren't queried.

**C. Update follower parser** (`packages/node/src/parsers/followers.ts`):

In the `parseFollower` function body, add 3 new field mappings to the `result` object, after the `address` line:

```ts

transactionIndex: raw.transaction_index ?? null,

These fields will be `undefined` when their `@include` is false (Hasura omits them), so `?? null` normalizes to the Zod schema's nullable type.

The `RawFollower` type derives from `GetFollowersQuery['follower'][number]` via codegen, so after running `pnpm codegen`, these fields will be available on `raw`.

import { buildBlockOrderSort, escapeLike, ... } from './utils';
```

**2. Update the `buildXOrderBy` switch statement:**

Remove the `'timestamp'` and `'blockNumber'` cases (they no longer exist in the sort field enum). Add `'newest'` and `'oldest'` cases:

```ts
case 'newest':
  return buildBlockOrderSort('desc');
case 'oldest':
  return buildBlockOrderSort('asc');
```

Note: `buildBlockOrderSort` returns `Record<string, Order_By>[]` which is structurally assignable to each domain's `*_Order_By[]` return type.

Keep all name-based sort cases unchanged (e.g., `'followerName'`, `'universalProfileName'`, `'digitalAssetName'`, etc.).

**3. Wire default sort** in each `fetchX` implementation function:

Change the `orderBy` assignment in the implementation body from:

```ts
const orderBy = buildXOrderBy(params.sort);
```

to:

```ts
const orderBy = buildXOrderBy(params.sort) ?? buildBlockOrderSort('desc');
```

This ensures that when no sort is specified, the service defaults to newest-first block ordering. `buildXOrderBy` returns `undefined` when `sort` is undefined, so the `??` fallback kicks in.
