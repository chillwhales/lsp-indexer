# Phase 9.12: Block-Ordered Sorting - Context

**Gathered:** 2026-02-26
**Status:** Ready for planning

<domain>
## Phase Boundary

All event domains sort by `blockNumber → transactionIndex → logIndex` instead of timestamp for deterministic ordering. Extract a shared sorting utility and apply across 4 event domains: Followers (9.5), Data Changed Events (9.10), Token ID Data Changed Events (9.10), Universal Receiver Events (9.11). Entity tables are NOT affected.

</domain>

<decisions>
## Implementation Decisions

### Sort field API surface

- Abstract away blockchain terminology completely — consumers never see `blockNumber`, `transactionIndex`, or `logIndex`
- Two self-contained sort fields: `'newest'` (block-order desc) and `'oldest'` (block-order asc) — direction and nulls are ignored when either is selected
- `transactionIndex` and `logIndex` are internal tiebreakers only — automatically appended when sorting by block order, never exposed to consumers
- Remove `'timestamp'` and `'blockNumber'` sort fields from all 4 event domain schemas
- Name-based sort fields (e.g., `'followerName'`, `'fromProfileName'`, `'digitalAssetName'`) remain unchanged with their existing `direction` + `nulls` behavior

### Default sort behavior

- Default (no sort specified) = newest first — applied at the service layer so every consumer (hooks, server actions, direct service calls) gets the correct default automatically
- No opt-out mechanism — omitting sort always produces block-order desc (newest first)
- `nulls` option silently ignored for `'newest'`/`'oldest'` — block-level fields are non-nullable in Hasura, so nulls has no effect. Keep the uniform `{ field, direction, nulls? }` sort shape unchanged
- All 4 event domains switch simultaneously — no phased rollout or special treatment for Followers (which currently has no block fields exposed)

### Playground sort controls

- Default selected sort field: `'newest'`
- Hide Direction and Nulls dropdowns when `'newest'` or `'oldest'` is selected — only show controls that have an effect
- Show Direction and Nulls dropdowns when a name-based sort is selected (where they matter)
- Keep "Ascending" / "Descending" direction labels unchanged — developer playground, no need for contextual labels

### Claude's Discretion

- Shared `buildBlockOrderSort()` utility implementation details (location, signature, internal structure)
- How to handle the sort schema type so `'newest'`/`'oldest'` fields can coexist with name sorts that use direction (union type, runtime ignore, or separate code paths)
- Playground conditional hide/show implementation approach

</decisions>

<specifics>
## Specific Ideas

- Consumer mental model is "newest first" / "oldest first" — not blockchain blocks or timestamps
- The `'newest'`/`'oldest'` fields are self-describing — no direction parameter needed to understand intent
- Playground should demonstrate real API behavior: if a control has no effect, hide it

</specifics>
