## Accomplishments

- Added `buildBlockOrderSort(direction)` utility to `packages/node/src/services/utils.ts` — 3-level tiebreaker (block_number, transaction_index, log_index) for deterministic event ordering
- Replaced `'timestamp'` and `'blockNumber'` sort fields with `'newest'` and `'oldest'` in all 4 event domain sort field schemas (Followers, DataChangedEvents, TokenIdDataChangedEvents, UniversalReceiverEvents)
- Added `blockNumber`, `transactionIndex`, `logIndex` fields to FollowerSchema, FollowerIncludeSchema, and FollowerScalarIncludeFieldMap
- Updated all 4 service sort builders to use buildBlockOrderSort for newest/oldest cases
- Updated fetchLatest\* functions to use 'newest' instead of 'timestamp'

## Decisions Made

- `buildBlockOrderSort` returns `Record<string, Order_By>[]` instead of a specific Hasura type — structural typing means it's assignable to all 4 `*_Order_By[]` types
- `direction` and `nulls` are silently ignored when `'newest'` or `'oldest'` is selected — these sort fields are self-describing
- Follower parser returns `null` for new block fields until the GraphQL document is updated in Plan 02
