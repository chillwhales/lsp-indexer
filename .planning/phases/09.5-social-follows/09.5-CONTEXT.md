# Phase 9.5: Social / Follows - Context

**Gathered:** 2026-02-22
**Status:** Ready for planning

<domain>
## Phase Boundary

Developer can use `useFollowers`, `useInfiniteFollowers`, `useFollowing`, `useInfiniteFollowing`, `useFollowCount`, `useIsFollowing` for LSP26 social graph data. Queries the `follower` state table (current active relationships), NOT the `follow`/`unfollow` event tables. Follows the validated 4-plan vertical-slice pattern.

</domain>

<decisions>
## Implementation Decisions

### Data source — `follower` state table, not event tables

- **Query the `follower` table** — represents current active follow relationships (one record per unique follower↔followed pair)
- **Do NOT expose `follow` or `unfollow` event tables** — those are historical event logs (multiple entries per pair) and are out of scope for this phase
- Follow/unfollow event history is deferred — could fit a future events-oriented phase if needed

### Follower type shape

- **Base fields (always present):** `followerAddress`, `followedAddress`
- **Include-toggled scalar fields:** `timestamp`, `address` (contract address)
- **Include-toggled relations:** `followerProfile` (ProfileIncludeSchema), `followedProfile` (ProfileIncludeSchema)
- **Domain type name:** `Follower` — matches Hasura table name, consistent with `Nft` → `nft`, `OwnedAsset` → `owned_asset`
- **Excluded from type:** `blockNumber`, `transactionIndex`, `logIndex`, `id` — indexer internals, not useful for social graph consumers

### Nested profile includes — sub-include objects

- `followerProfile: ProfileIncludeSchema` — sub-include object for full control over which profile fields to fetch for the follower
- `followedProfile: ProfileIncludeSchema` — sub-include object for the followed profile
- Matches established pattern: NFTs use `collection: DigitalAssetIncludeSchema`, `holder: ProfileIncludeSchema`
- Consumers can request different fields for each side: `include: { followerProfile: { name: true }, followedProfile: { name: true, avatar: true } }`

### Hook set — 6 hooks

1. `useFollowers(address)` — paginated list of followers for an address (query `follower` table where `followed_address = address`)
2. `useInfiniteFollowers(address)` — infinite scroll variant
3. `useFollowing(address)` — paginated list of who an address follows (query `follower` table where `follower_address = address`)
4. `useInfiniteFollowing(address)` — infinite scroll variant
5. `useFollowCount(address)` — returns `{ followerCount: number, followingCount: number }` via two aggregates on the `follower` table
6. `useIsFollowing({ followerAddress, followedAddress })` — boolean check if one address follows another (query `follower` table with both addresses, check if record exists)

### Filter fields

- `followerAddress` — `_ilike` case-insensitive match (established pattern)
- `followedAddress` — `_ilike` case-insensitive match
- `followerName` — nested profile name search via `followerUniversalProfile.lsp3Profile.name._ilike`
- `followedName` — nested profile name search via `followedUniversalProfile.lsp3Profile.name._ilike`
- `timestampFrom` — ISO string, maps to `timestamp: { _gte }` — optional lower bound
- `timestampTo` — ISO string, maps to `timestamp: { _lte }` — optional upper bound
- All string fields use `escapeLike` (established pattern)

### Sort fields

- `timestamp` — primary sort ("newest followers first")
- `followerAddress` — alphabetical by follower address
- `followedAddress` — alphabetical by followed address
- `followerName` — nested sort via `followerUniversalProfile.lsp3Profile.name` with `orderDir()` + `SortNulls` (established pattern, matches `digitalAssetName` nested sort in owned-assets)
- `followedName` — nested sort via `followedUniversalProfile.lsp3Profile.name`

### Playground page — single page, 6 tabs

- Single page at `/followers` with 6 tabs — one per hook
- **Tabs:** "Followers" / "Infinite Followers" / "Following" / "Infinite Following" / "Count" / "Is Following"
- Each tab has its own independent inputs, filters, and sorts — no shared address input across tabs
- **Count tab:** single address input → displays `{ followerCount, followingCount }`
- **Is Following tab:** two address inputs (`followerAddress`, `followedAddress`) → displays boolean result

### Follower card display

- **Always visible:** `followerAddress`, `followedAddress`, `timestamp`
- **Collapsible section 1:** Follower Profile — shown when `include.followerProfile` is provided, uses ghost Button trigger with ChevronDown (established pattern)
- **Collapsible section 2:** Followed Profile — shown when `include.followedProfile` is provided, same collapsible pattern
- Full addresses, no truncation (established `break-all` CSS wrapping preference)
- `'key' in obj` field-presence checks for conditional rendering (established pattern from DX-04)

### Conditional include types (DX-04 from the start)

- This is the first domain built WITH conditional include types from the start (not retrofitted)
- `FollowerResult<I>` with `ResolveFollowerProfile<I>` and `ResolveFollowedProfile<I>` intersection types
- `stripExcluded` at service boundary with base fields `['followerAddress', 'followedAddress']`
- `as FollowerResult<I>` cast at service boundary (established pattern)
- Recursive nested stripping via `parseProfile(raw, include?.followerProfile)` delegation

### Claude's Discretion

- Exact GraphQL document field selection and variable naming
- Parser internal implementation details
- Query key factory structure (follow established patterns from other domains)
- Hook parameter naming and overload design beyond what's specified
- Playground page layout, spacing, and styling details
- Error state handling in playground
- Loading skeleton design
- How `useIsFollowing` internally queries (single record check vs aggregate count)

</decisions>

<specifics>
## Specific Ideas

- "Follow and Unfollow are smart contract events... Follower entities are specific follow relation between follower address and followed address" — use `follower` state table, not event tables
- Every hook gets its own playground tab — 6 hooks, 6 tabs, no combining
- Card layout: addresses + timestamp always visible, two collapsible profile sections below — consistent with NFT card pattern (holder profile + collection as collapsible sections)
- `useIsFollowing` takes two named address fields matching the type: `{ followerAddress, followedAddress }` — no ambiguity about direction

</specifics>

<deferred>
## Deferred Ideas

- **Follow/unfollow event history** — `follow` and `unfollow` event tables could be exposed as historical data in a future events-oriented phase. Not needed for current social graph queries.

</deferred>
