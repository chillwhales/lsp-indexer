**Gathered:** 2026-02-20
**Status:** Ready for planning

<domain>
## Phase Boundary

Developer can use `useOwnedAssets`, `useOwnedTokens`, `useInfiniteOwnedAssets`, `useInfiniteOwnedTokens` for LSP7 fungible + LSP8 NFT ownership data. Two Hasura tables (`owned_asset`, `owned_token`) treated as two separate domains within one phase. Follows the validated 4-plan vertical-slice pattern.

</domain>

<decisions>
## Implementation Decisions

### Playground page organization

- Two separate playground pages: `/owned-assets` and `/owned-tokens` — fully independent, not tabs
- Each page is its own domain with independent filters, sorts, and cards
- Both ship in the same phase (9.3) but are treated as distinct domains in code (separate files for types, documents, parsers, services, keys, hooks, actions)

### Owned assets card display

- Core fields: owner + asset address + balance
- Include toggles for digital asset details AND universal profile details (same nested include pattern as NFTs collection)
- Timestamp shown as secondary info (relative time, e.g. "2 days ago")

### Owned tokens card display

- Core fields: owner + asset address + token ID
- Include toggles for NFT details (which already bundles digital asset data via `parseDigitalAsset` reuse) AND universal profile — three levels of nested data through includes
- Timestamp shown as secondary info (relative time)

### Balance / big number handling

- Parser returns `bigint` for balance field — not formatted, consumer handles formatting
- Audit `totalSupply` in digital assets domain to ensure it also returns `bigint` (consistency across all uint256 values)
- Test UI formats balance when digital asset decimals are available (reuse `formatTokenAmount` utils), raw `bigint` otherwise

### Owned assets filters

- All filterable fields exposed: owner, address, balance, timestamp, digital_asset_id, universal_profile_id
- Address fields use `_ilike` for case-insensitivity (established pattern)
- String fields get `escapeLike` (established pattern)

### Owned tokens filters

- All filterable fields exposed: owner, address, token_id, timestamp, digital_asset_id, nft_id, owned_asset_id, universal_profile_id
- Same `_ilike` and `escapeLike` patterns as owned assets

### Owned assets sorts

- All direct columns: balance, timestamp, address, owner, block
- Nested sorts: digitalAsset (name, etc.), universalProfile, tokenIds_aggregate (count)
- All use `orderDir()` helper with `SortNulls` (established pattern)

### Owned tokens sorts

- All direct columns: address, block, owner, timestamp, token_id
- Nested sorts: digitalAsset, nft, ownedAsset, universalProfile
- Same `orderDir()` helper pattern

### Claude's Discretion

- Exact GraphQL document field selection and variable naming
- Parser internal implementation details
- Query key factory structure (follow established patterns)
- Hook parameter naming and overload design
- Playground page layout and spacing details
- Error state handling in playground

</decisions>

<specifics>
## Specific Ideas

- "Include details for both digital asset and profile similar to how we did in NFTs" — the nested include toggle pattern from Phase 9.2 is the reference
- "NFT include will already include digital asset stuff, so really we get all 3" — owned tokens get NFT → digital asset → profile chain
- "Let user handle formatting if he needs to, return bigint" — library returns raw bigint, only test UI formats
- Two separate domains but one phase — code separation is real, delivery is grouped

</specifics>
