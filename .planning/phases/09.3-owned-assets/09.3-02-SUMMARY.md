# Phase 9.3 Plan 02: Owned Asset/Token Parsers, Services & Keys Summary

**One-liner:** Query key factories, BigInt-balance parsers, and filter/sort/include service layer for both OwnedAsset and OwnedToken domains with cross-domain parser reuse.

## Performance

- **Duration:** ~4.5 minutes
- **Tasks:** 2/2 completed
- **Commits:** 2 task commits
- **Deviations:** 1 (type cast fix for cross-domain parser sub-selections)

## Accomplishments

1. **Query key factories** — `ownedAssetKeys` and `ownedTokenKeys` with TkDodo hierarchical pattern: `all` → `details`/`detail` → `lists`/`list` → `infinites`/`infinite`. Separate list/infinite namespaces prevent TanStack Query cache corruption.

2. **Parsers with BigInt balance** — `parseOwnedAsset` converts Hasura `numeric` (string) balance to JavaScript `bigint` via `BigInt()` for uint256 precision. `parseOwnedToken` maps `token_id` → `tokenId` (snake_case → camelCase). Both parsers reuse `parseDigitalAsset`, `parseNft`, and `parseProfile` for nested relations.

3. **Service functions** — `fetchOwnedAsset`/`fetchOwnedAssets` and `fetchOwnedToken`/`fetchOwnedTokens` translate developer-friendly filter/sort/include params to Hasura GraphQL variables:

   - All 4 owned asset filter fields + 7 owned token filter fields use `_ilike` + `escapeLike`
   - All sort fields use `orderDir()` with SortNulls
   - Nested sorts: `digitalAssetName` → `{ digitalAsset: { lsp4TokenName: { value: dir } } }`, `tokenIdCount` → `{ tokenIds_aggregate: { count: dir } }`
   - `buildDigitalAssetIncludeVars` reused for nested DA sub-includes in both services

4. **Barrel exports** — All 6 new modules exported from `packages/node/src/index.ts` following explicit named export pattern.

## Decisions Made

1. **Cross-domain parser `as any` casts:** Nested sub-selections in owned_asset/owned_token GraphQL documents omit fields like `id` that primary domain parsers expect (e.g., `universalProfile` in owned_asset doesn't select `id`, but `parseProfile` expects `RawProfile` which requires `id`). Using `as any` cast is safe because all parsers use optional chaining and gracefully handle missing fields. This is the same pattern used whenever a sub-selection provides fewer fields than the primary query type.

2. **DA include vars reused directly (not prefixed):** Unlike NFT service which prefixes DA vars as `includeCollection*`, the owned asset/token services pass DA vars directly since the owned_asset/owned_token GraphQL documents use the same `includeName`, `includeSymbol`, etc. variable names at the top level.
