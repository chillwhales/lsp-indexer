# Phase 5.3: Entity Upsert Pattern Standardization - Research

**Researched:** 2026-02-13
**Domain:** Handler entity lifecycle management (unified resolve pattern)
**Confidence:** HIGH

## Summary

Comprehensive audit of all 29 V2 handlers reveals **4 different ad-hoc patterns** for the same operation: "check if entity already exists before creating/updating." This inconsistency has produced **3 confirmed bugs** and **2 cross-batch gaps**. More importantly, there is no single recognizable pattern — a developer reading any handler has to reverse-engineer what entity lifecycle strategy is being used.

This phase introduces two helper functions (`resolveEntity` and `resolveEntities`) and a single canonical code shape: **resolve → spread → override → addEntity**. Every handler that creates or updates entities that might already exist gets refactored to use this exact pattern. The goal is that when you see `...existing` in a constructor, you immediately know the handler is safe. When you don't, you know it's a create-only handler.

**Primary recommendation:** Create `resolveEntity<T>()` and `resolveEntities<T>()` helpers. Refactor all 13 handlers that do any form of "entity might exist" lookup to use the same recognizable pattern. Remove `mergeEntitiesFromBatchAndDb` (replaced by `resolveEntities`). Keep `setPersistHint` as an optional belt-and-suspenders for the Index+Map merge handlers.

## The Problem: 4 Patterns for 1 Operation

| Pattern                                                                                   | Handlers                                              | Recognizable?           |
| ----------------------------------------------------------------------------------------- | ----------------------------------------------------- | ----------------------- |
| `mergeEntitiesFromBatchAndDb()` + manual field merge + `addEntity()` + `setPersistHint()` | lsp4Creators, lsp12IssuedAssets, lsp6Controllers, nft | ❌ 4 mechanisms layered |
| Manual `store.findBy()` + batch `.get()` + `?? null` per FK                               | totalSupply, ownedAssets                              | ❌ Custom per-handler   |

| No check at all, blind `new Entity({...null FKs...})`                                     | chillClaimed, orbsClaimed                             | ❌ Wipes FKs            |

**None of these is a "pattern." They're all bespoke solutions.**

## The Solution: One Pattern Everywhere

### The Helpers

```typescript
// Single entity lookup: batch → DB → null
async function resolveEntity<T extends Entity>(
  store: Store,
  batchCtx: IBatchContext,
  entityType: string,
  entityClass: EntityConstructor<T>,
  id: string,
): Promise<T | null>;

// Bulk entity lookup: batch → DB → Map (efficient for N entities)
async function resolveEntities<T extends Entity>(
  store: Store,
  batchCtx: IBatchContext,
  entityType: string,
  entityClass: EntityConstructor<T>,
  ids: string[],
): Promise<Map<string, T>>;
```

These replace `mergeEntitiesFromBatchAndDb` with a cleaner name and identical semantics. The old function is deleted; all call sites migrate to `resolveEntities`.

### The Canonical Code Shape

Every handler that updates an entity that might already exist uses this exact shape:

```typescript
// 1. RESOLVE — one function, always batch → DB → null
const existing = await resolveEntity(hctx.store, hctx.batchCtx, TYPE, EntityClass, id);

// 2. BUILD — spread existing (preserves ALL fields including FKs), override what changed
const entity = new EntityClass({
  ...existing, // no-op if null; preserves ALL fields + FKs if not
  id, // always explicit
  field1: newValue1, // override only what this handler owns
  field2: newValue2,
});

// 3. COMMIT — always addEntity
hctx.batchCtx.addEntity(TYPE, id, entity);
```

**The visual signature:** When you see `...existing` in a constructor, the handler is safe. When you see `new Entity({ id, ..., digitalAsset: null, nft: null })` without a prior resolve, it's either a create-only path (fine) or a bug.

### Bulk Variant (for handlers processing many entities)

```typescript
// 1. COLLECT IDs
const ids = transfers.map((t) => generateId(t));

// 2. RESOLVE ALL — one DB query for all
const existingMap = await resolveEntities(hctx.store, hctx.batchCtx, TYPE, EntityClass, ids);

// 3. For each entity:
const existing = existingMap.get(id) ?? null;
const entity = new EntityClass({
  ...existing,
  id,
  value: newValue,
});
hctx.batchCtx.addEntity(TYPE, id, entity);
```

## Architecture Details

### resolveEntity Implementation

```typescript
/**
 * Resolve a single entity from batch or database.
 *
 * The standard lookup order:
 *   1. Check BatchContext (current batch, in-memory)
 *   2. Check database (previous batches, persisted)
 *   3. Return null if not found
 *
 * This is THE way to look up existing entities in handlers.
 * Every handler that updates an entity that might already exist
 * MUST use this function (or resolveEntities for bulk).
 */
export async function resolveEntity<T extends Entity>(
  store: Store,
  batchCtx: IBatchContext,
  entityType: string,
  entityClass: EntityConstructor<T>,
  id: string,
): Promise<T | null> {
  // 1. Check batch first (current batch)
  const batchEntity = batchCtx.getEntities<T>(entityType).get(id);
  if (batchEntity) return batchEntity;

  // 2. Check database (previous batches)
  const dbEntity = await store.findOneBy(entityClass, { id } as FindOptionsWhere<T>);
  return dbEntity ?? null;
}
```

### resolveEntities Implementation

```typescript
/**
 * Resolve multiple entities from batch and database.
 *
 * Efficient bulk variant: single DB query for all IDs not found in batch.
 * Returns a merged Map with batch entities taking priority over DB entities.
 *
 * This replaces the old mergeEntitiesFromBatchAndDb() with an identical
 * implementation but a clearer name that matches the resolveEntity API.
 */
export async function resolveEntities<T extends Entity>(
  store: Store,
  batchCtx: IBatchContext,
  entityType: string,
  entityClass: EntityConstructor<T>,
  ids: string[],
): Promise<Map<string, T>> {
  // 1. Get all from batch
  const batchEntities = batchCtx.getEntities<T>(entityType);
  const merged = new Map<string, T>();

  // Add batch entities for requested IDs
  for (const id of ids) {
    const batchEntity = batchEntities.get(id);
    if (batchEntity) merged.set(id, batchEntity);
  }

  // 2. Query DB for IDs not in batch
  const idsNotInBatch = ids.filter((id) => !merged.has(id));
  if (idsNotInBatch.length > 0) {
    const dbEntities = await store.findBy(entityClass, {
      id: In(idsNotInBatch),
    } as FindOptionsWhere<T>);
    for (const entity of dbEntities) {
      merged.set(entity.id, entity);
    }
  }

  return merged;
}
```

### Migration: mergeEntitiesFromBatchAndDb → resolveEntities

`resolveEntities` is functionally identical to `mergeEntitiesFromBatchAndDb` — same semantics, better name, same file location (`handlerHelpers.ts`). The migration is:

1. Add `resolveEntity` and `resolveEntities` to `handlerHelpers.ts`
2. Update all imports from `mergeEntitiesFromBatchAndDb` → `resolveEntities`
3. Delete the old function
4. Update the handler code to use spread pattern instead of manual field mutation

### The Spread Pattern in Detail

**Why spread works for FK preservation:**

TypeORM entities are plain objects with `Object.assign(this, props)` in the constructor. When you spread an existing entity into a new constructor, ALL fields are carried over — including FK references. Then you override only the fields your handler owns:

```typescript
// existing = { id, address, tokenId, value: 0, digitalAsset: { id: '0x...' }, nft: { id: '...' } }

const entity = new OrbLevel({
  ...existing, // carries: id, address, tokenId, digitalAsset, nft, value
  value: newValue, // overrides ONLY value
});
// Result: digitalAsset and nft are PRESERVED from existing
```

**For create case (existing is null):**

```typescript
const existing = await resolveEntity(...);

// When existing is null, spread is a no-op:
const entity = new OrbLevel({
  ...existing,        // ...null → no-op (Object spread of null/undefined is empty)
  id,
  address,
  tokenId,
  value: 0,
  digitalAsset: null,
  nft: null,
});
```

Wait — `...null` doesn't work in TypeScript. We need to handle the null case. Two options:

**Option A: Conditional spread**

```typescript
const entity = new OrbLevel({
  ...(existing ?? {}),
  id,
  value: newValue,
  // ... other fields for create case
});
```

**Option B: Ternary**

```typescript
const entity = existing
  ? new OrbLevel({ ...existing, value: newValue })
  : new OrbLevel({ id, address, tokenId, value: newValue, digitalAsset: null, nft: null });
```

**Recommendation: Option A** — single code path, less branching, still recognizable. `...(existing ?? {})` is a clear "spread if exists, empty if not" idiom.

Actually, let me reconsider. `...null` DOES work in JavaScript/TypeScript:

```typescript
const obj = { ...null }; // → {}
const obj2 = { ...undefined }; // → {}
```

This is valid ES2018+ behavior. So `...existing` when existing is null simply produces `{}`. **No special handling needed.** The pattern is just:

```typescript
const existing = await resolveEntity(...) // T | null

const entity = new OrbLevel({
  ...existing,     // {} if null, all fields if not
  id,              // always explicit (required for create case)
  address,         // always explicit (required for create case)
  tokenId,         // always explicit
  value: newValue, // always explicit (the thing we're updating)
  digitalAsset: existing?.digitalAsset ?? null,  // FK: preserve if exists, null if create
  nft: existing?.nft ?? null,                    // FK: preserve if exists, null if create
});
```

Hmm, but now FKs need explicit handling again. The spread carries them, but then we override them with the explicit fields. Let me think...

Actually, the issue is for the **create case**: when `existing` is null, spread gives `{}`, and then we MUST provide all required fields including FKs as `null`. For the **update case**: when `existing` has values, spread carries them, and then we DON'T want to override FKs.

**The cleanest approach:**

```typescript
const existing = await resolveEntity(hctx.store, hctx.batchCtx, TYPE, Class, id);

if (existing) {
  // UPDATE: spread preserves all fields, override only what changed
  const entity = new EntityClass({ ...existing, value: newValue });
  hctx.batchCtx.addEntity(TYPE, id, entity);
} else {
  // CREATE: all fields explicit, FKs null
  const entity = new EntityClass({
    id, address, tokenId,
    value: newValue,
    digitalAsset: null,
    nft: null,
  });
  hctx.batchCtx.addEntity(TYPE, id, entity);
  // Queue enrichment (only on create — existing already enriched)
  hctx.batchCtx.queueEnrichment(...);
}
```

This is cleaner than trying to merge create+update into one expression. The branching is explicit, the intent is clear, and enrichment only happens on create (not on every update).

**But** this means enrichment isn't re-queued for existing entities from DB that may have lost their FKs. In practice, if the entity is in the DB, enrichment already ran for it. The only exception is the chillClaimed/orbsClaimed case where Phase 2 WIPES FKs — but with spread, they're preserved.

**Final recommended pattern:**

```typescript
const existing = await resolveEntity(hctx.store, hctx.batchCtx, TYPE, EntityClass, id);

const entity = new EntityClass({
  ...existing,                                    // preserves ALL fields if update; {} if create
  id,                                             // always explicit
  address,                                        // always explicit
  value: newValue,                                // override what changed
  digitalAsset: existing?.digitalAsset ?? null,   // FK: preserve or null
  nft: existing?.nft ?? null,                     // FK: preserve or null
});

hctx.batchCtx.addEntity(TYPE, id, entity);

// Queue enrichment (idempotent — pipeline deduplicates)
hctx.batchCtx.queueEnrichment(...);
```

FKs are explicit on every path with `existing?.fk ?? null`. This is the safest, most readable approach.

## Common Pitfalls

### Pitfall 1: Forgetting `...existing` (the whole point of this phase)

**Symptom:** `new Entity({ ..., digitalAsset: null, nft: null })` where entity might exist
**Prevention:** `resolveEntity` → `...existing` is the standard. Lint rule could catch `digitalAsset: null` without a prior `resolveEntity` call.

### Pitfall 2: Not calling `addEntity()` after modifying a DB entity

**Symptom:** Data merged in-memory but never persisted (lsp5ReceivedAssets bug)
**Prevention:** The pattern always calls `addEntity()` — there's no "modify in place and return" path.

### Pitfall 3: Using `mergeEntitiesFromBatchAndDb` instead of `resolveEntities`

**Symptom:** Old function name in new code
**Prevention:** Delete the old function. Compiler errors force migration.
