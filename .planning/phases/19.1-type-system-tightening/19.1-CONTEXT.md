# Phase 19.1 Context: Type System Tightening

## Background

Phase 19 (Block Ordering) widened the `Entity` base type from `{ id: string }` to
`{ id: string; blockNumber: number; transactionIndex: number; logIndex: number }`.
This makes `Entity` a meaningful base type that reflects the reality: every entity
in the system carries block ordering fields.

During PR #284 review, we identified several cascading improvements that should
follow from this change but were too large to include in the block ordering PR:

1. **`BatchContext.getEntities` returns `Map<string, Entity>` instead of generic `<T>`** —
   The old `getEntities<T>()` was an unchecked cast disguised as type safety. The bag
   is type-erased (stores by string key), so there's no compile-time proof that
   `'LSP7Transfer'` maps to `Transfer`. Handlers should get `Map<string, Entity>` and
   cast at the point of use, where they actually know what they stored.

2. **`BatchContext.addEntity` accepts `Entity` instead of `unknown`** — With the
   widened `Entity` type, we can enforce that only proper entities enter the bag.

3. **Pipeline removes `as Entity[]` casts** — Since `getEntities()` returns `Entity`
   directly, the pipeline no longer needs to cast `unknown[]` to `Entity[]`.

4. **Handler test fixes** — All handler tests need block fields on test entity objects
   and updated mock expectations (e.g., `blockNumber: 100` from test factories instead
   of `0`, `isDebug` mock on logger).

5. **Integration test fixes** — `multi-event.json` fixture has a malformed DataChanged
   log (raw UTF-8 hex instead of ABI-encoded bytes), and `createMockVerifyFn` needs
   the 5th `blockPositionByAddress` parameter.

## Scope

~29 handler files + ~5 core files + ~12 test files. All changes are mechanical:
remove type parameter from `getEntities` calls, add `as Map<string, SpecificType>`
cast at the handler level where the handler knows what type it stored.
