# Phase 9.8: Issued Assets - Context

**Gathered:** 2026-02-23
**Status:** Ready for planning

<domain>
## Phase Boundary

Developer can use `useIssuedAssets`, `useInfiniteIssuedAssets` for LSP12 issued asset data — querying which digital assets were issued by which addresses/profiles. Follows the established 4-plan vertical-slice pattern (types+docs → parsers+services+keys → hooks+actions+wiring → playground+verification). No singular `useIssuedAsset` hook (issued asset records are a join table with no natural key — developers query by filter). Nearly identical structure to Phase 9.7 (Creators) — same decisions apply, adapted for the `lsp12_issued_asset` Hasura table.

</domain>

<decisions>
## Implementation Decisions

### Domain naming

- Domain name: **"IssuedAsset"**
- Types: `IssuedAsset`, `IssuedAssetInclude`, `IssuedAssetResult<I>`, `PartialIssuedAsset`
- Hooks: `useIssuedAssets`, `useInfiniteIssuedAssets`
- Query keys: `issuedAssetKeys`
- Files: `issued-assets.ts` across all packages
- No singular `useIssuedAsset` hook — no natural key exists (opaque Hasura ID only)

### Nested relation includes

- `issuerProfile` (from Hasura `universalProfile`) → full sub-include support via `ProfileInclude` / `ProfileResult<I>`
  - `include: { issuerProfile: true }` fetches all profile fields
  - `include: { issuerProfile: { name: true, avatar: true } }` narrows to specified fields
  - Reuses `ProfileIncludeSchema`, `buildProfileIncludeVars`, `parseProfile`
- `digitalAsset` (from Hasura `issuedAsset`) → full sub-include support via `DigitalAssetInclude` / `DigitalAssetResult<I>`
  - `include: { digitalAsset: true }` fetches all DA fields
  - `include: { digitalAsset: { name: true, symbol: true } }` narrows to specified fields
  - Reuses `DigitalAssetIncludeSchema`, `buildDigitalAssetIncludeVars`, `parseDigitalAsset`

### Base fields (always present)

- `issuerAddress` (from `address`) — who issued the asset
- `assetAddress` (from `asset_address`) — the digital asset issued
- These are the composite natural key of the relationship

### Includable scalar fields

- `arrayIndex` (from `array_index`: `numeric`) — position in LSP12 issued assets array
- `interfaceId` (from `interface_id`: `String`) — ERC165 interface ID
- `timestamp` (from `timestamp`: `timestamptz`) — when indexed

### Filter fields (7 total)

- `issuerAddress` → `address._ilike` (case-insensitive)
- `assetAddress` → `asset_address._ilike` (case-insensitive)
- `interfaceId` → `interface_id._ilike` (case-insensitive)
- `issuerName` → `universalProfile.lsp3Profile.name._ilike` (nested, same pattern as creators' `creatorName`)
- `digitalAssetName` → `issuedAsset.lsp4TokenName.name._ilike` (nested, same pattern as creators)
- `timestampFrom` → `timestamp: { _gte }` (range lower bound)
- `timestampTo` → `timestamp: { _lte }` (range upper bound)

### Sort fields (6 total)

- `timestamp` → `timestamp` (default sort: descending)
- `issuerAddress` → `address`
- `assetAddress` → `asset_address`
- `arrayIndex` → `array_index`
- `issuerName` → `universalProfile.lsp3Profile.name` (nested sort)
- `digitalAssetName` → `issuedAsset.lsp4TokenName.name` (nested sort)
- Default sort: `timestamp` descending
- All sorts support `SortNulls` via `orderDir()` helper

### Playground card layout

- **Top-level row:** All scalar fields — `issuerAddress`, `assetAddress`, `arrayIndex`, `interfaceId`, `timestamp` (render via `'key' in obj` guards)
- **Collapsible 1:** Issuer Profile (when `issuerProfile` included)
- **Collapsible 2:** Digital Asset (when `digitalAsset` included)
- Collapsibles use ghost Button + ChevronDown pattern (matches FollowerCard / CreatorCard)
- Plus standard RawJsonToggle

### Playground page

- Two tabs: **List** (`useIssuedAssets` with filters/sort/include toggles) + **Infinite** (`useInfiniteIssuedAssets`)
- Route: `apps/test/src/app/issued-assets/page.tsx`
- Add to nav sidebar

### Claude's Discretion

- Exact Hasura field → camelCase mapping details beyond what's specified
- IssuedAssetCard component internal styling/spacing
- Include toggle UI layout in playground
- Error state handling in playground

</decisions>

<specifics>
## Specific Ideas

- Nearly identical to Phase 9.7 (Creators) — same card layout, same include patterns, same filter/sort structure
- Only differences: Hasura table name (`lsp12_issued_asset` vs `lsp4_creator`), field mappings (`address`→`issuerAddress` vs `creator_address`→`creatorAddress`), nested relation names (`universalProfile`→`issuerProfile` vs `creatorProfile`)
- Card layout modeled after FollowerCard / CreatorCard — scalars first, collapsible nested relations below
- Both nested relations support boolean `true` (all fields) OR object (narrowed fields)
- Follows the 3-overload `<const I>` generic pattern established in Phase 9.6

</specifics>
