# Phase 9.10: Data Changed Events - Context

**Gathered:** 2026-02-25
**Status:** Ready for planning

<domain>
## Phase Boundary

Developer can use `useDataChangedEvents`, `useInfiniteDataChangedEvents` for ERC725 data change event history, AND `useTokenIdDataChangedEvents`, `useInfiniteTokenIdDataChangedEvents` for per-token data change events. Two distinct domains implemented together in one phase — both follow the established 4-plan vertical-slice pattern.

Hasura tables: `data_changed` (contract-level, has `universalProfile` + `digitalAsset` relations) and `token_id_data_changed` (token-level, has `nft` + `digitalAsset` relations, plus `token_id` field).

</domain>

<decisions>
## Implementation Decisions

### Domain scope

- Both `data_changed` AND `token_id_data_changed` tables exposed as two distinct domains in this single phase
- They are ~90% identical in structure — follow the same established domain creation pattern from previous sub-phases
- `data_changed` hooks: `useDataChangedEvents`, `useInfiniteDataChangedEvents`
- `token_id_data_changed` hooks: `useTokenIdDataChangedEvents`, `useInfiniteTokenIdDataChangedEvents`
- No singular hooks for either domain (event tables, no natural single-entity key)

### Data key name resolution

- Parser returns raw hex `dataKey` always, plus a resolved `dataKeyName: string | null` for known ERC725Y keys
- Known keys sourced from official LUKSO packages (runtime dependencies of `@lsp-indexer/node`):
  - `@lukso/lsp1-contracts` (LSP1DataKeys)
  - `@lukso/lsp3-contracts` (LSP3DataKeys)
  - `@lukso/lsp4-contracts` (LSP4DataKeys)
  - `@lukso/lsp5-contracts` (LSP5DataKeys)
  - `@lukso/lsp6-contracts` (LSP6DataKeys)
  - `@lukso/lsp8-contracts` (LSP8DataKeys)
  - `@lukso/lsp9-contracts` (LSP9DataKeys)
  - `@lukso/lsp10-contracts` (LSP10DataKeys)
  - `@lukso/lsp12-contracts` (LSP12DataKeys)
  - `@lukso/lsp17contractextension-contracts` (LSP17DataKeys)
- Unknown keys return `dataKeyName: null`

### Data value display

- `dataValue` returned as raw hex only — no decoding
- ERC725Y value decoding (interpreting values by key type) is deferred

### Filters — `data_changed`

- 6 filter fields:
  - `address` (ilike) — the emitting contract address (is either a UP or DA, cannot be both)
  - `dataKey` (ilike)
  - `timestamp` (range)
  - `blockNumber` (range)
  - UP name (nested: `universalProfile.lsp3Profile.name.value._ilike`)
  - DA name (nested: `digitalAsset.lsp4TokenName.name._ilike`)

### Filters — `token_id_data_changed`

- 7 filter fields:
  - `address` (ilike)
  - `dataKey` (ilike)
  - `tokenId` (ilike)
  - `timestamp` (range)
  - `blockNumber` (range)
  - DA name (nested: `digitalAsset.lsp4TokenName.name._ilike`)
  - NFT name (nested through `nft` relation)

### Sorts — `data_changed`

- 4 sort fields: `timestamp` (default desc), `blockNumber`, UP name, DA name

### Sorts — `token_id_data_changed`

- 4 sort fields: `timestamp` (default desc), `blockNumber`, DA name, NFT name

### Claude's Discretion

- Playground card layout and section grouping for event data
- How to display `dataKeyName` alongside `dataKey` in the card (label + hex, tooltip, etc.)
- Conditional include structure for nested relations (`universalProfile`, `digitalAsset`, `nft`)
- Parser field naming (camelCase mapping from Hasura snake_case)
- How to structure the data key resolver (single utility, per-domain lookup, reverse map, etc.)

</decisions>

<specifics>
## Specific Ideas

- `address` field is the emitting contract — it's inherently either a Universal Profile or a Digital Asset, never both. Don't expose separate `universalProfileId` / `digitalAssetId` filters — `address` covers both.
- The two domains are ~90% similar. Follow the exact same pattern as previous domains (creators, issued assets, encrypted assets, etc.)
- `@lukso/lsp4-contracts` was previously removed as a dep (only needed 3 integer values). This time the use case is stronger — 10 packages provide comprehensive ERC725Y key→name mappings at runtime.

</specifics>

<deferred>
## Deferred Ideas

- ERC725Y value decoding — interpreting `dataValue` based on key type (JSONURL, address, uint256, bytes[], etc.). Requires understanding each LSP standard's encoding rules. Could be a future enhancement or utility.

</deferred>
